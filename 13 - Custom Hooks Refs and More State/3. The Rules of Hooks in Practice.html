1) The Rule We‚Äôre About to Break

‚úÖ Main Rule (Rule #1)
Hooks must be called at the top level of a component (not inside if, loops, nested functions, or after ‚Äúearly returns‚Äù).

Why does React care? Because React tracks hooks primarily by call order (Hook #1, Hook #2, Hook #3, ‚Ä¶). If the order changes between renders, React can‚Äôt reliably match each hook to its stored value.

üîç Quick mental model
Think of hooks like seats in a row. If ‚ÄúSeat #4‚Äù used to be a useEffect, but later becomes a useState, everyone gets the wrong seat assignment. Chaos. üß®

3) Experiment #1 ‚Äî Calling useState Conditionally ‚ùå

Goal: Create a scenario where the hook list changes between renders.

‚ö†Ô∏è Don‚Äôt do this in real code
This is purely to understand the rule.

‚ùå Bad example (conditional hook)

// (Shown for learning only)

  const [isTop, setIsTop] = useState(true);
}

What happens (step-by-step)

  • First render: imdbRating might be undefined (movie not fetched yet).
  • The condition imdbRating > 8 is false, so the extra useState is not called.
  • Later, data arrives ‚Üí component re-renders ‚Üí now imdbRating becomes something like 8.8.
  • Condition becomes true ‚Üí suddenly React calls a new useState in the middle of the hook list.
  • React detects that the hook order changed and throws an error/warning.
üßæ What the React error is telling you
‚ÄúReact has detected a change in the order of Hooks called by this component.‚Äù That means: between renders, Hook #4 used to be something else (e.g., an effect), but now Hook #4 is a new state hook.

‚úÖ Correct approach

Always declare hooks at the top level. If you need ‚Äúconditional behavior,‚Äù make the behavior conditional, not the hook call. 

// ‚úÖ RIGHT: Declare hooks at top-level

  // ‚úÖ Conditional behavior is fine
  if (!imdbRating) return;
  setIsTop(Number(imdbRating) > 8);
}, [imdbRating]);
‚úÖ Rule-friendly pattern
‚ÄúHooks are unconditional. Logic inside them can be conditional.‚Äù

4) Experiment #2 ‚Äî Early Return That Skips Hooks ‚ùå

Another ‚Äúsneaky‚Äù way to break the hook order is returning early before all hooks run. This acts like a conditional path: sometimes hooks run, sometimes they don‚Äôt.

‚ùå Bad example (early return)

  // ...

Why it fails

  • Some renders return early ‚Üí fewer hooks are called.
  • Other renders don‚Äôt return early ‚Üí more hooks are called.

‚úÖ Correct approach

If you need an early return, do it after all hooks, or reorganize the component so hooks always run.

  // ...
}, []);

  return <p>Greatest ever</p>;
}

6) Quick Recap (Memory Anchors)

‚úÖ Hooks are tracked by order (Hook #1, Hook #2, Hook #3‚Ä¶).

‚ùå Conditional hooks (if, loops, nested functions) break order.

‚ùå Early returns can also break order if they skip hooks.

‚úÖ Keep hooks at top-level; make behavior conditional instead.
