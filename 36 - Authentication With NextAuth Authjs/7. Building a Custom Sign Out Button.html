Key idea üôÇ

A server action can be invoked from a client component and it will still run only on the server. This lets you keep the auth flow server-side (cleaner + often better UX).

üß© Why this pattern is needed

Problem: The Sign Out button is inside a client component

Your side navigation is a client component (it has interactivity/state). Therefore, a Sign Out button inside it is also client-side.

Temptation: Use onClick directly

In theory you could wire an onClick to some function, but the course uses the same server-action form pattern as the Sign In button.

Important üôÇ‚Äç‚ÜïÔ∏è

With Auth.js, signOut() is a server-side helper function you typically want to execute on the server. Server actions are the clean bridge for that.

üõ† Step-by-step implementation

1) Ensure Auth.js exports signOut

In your auth setup file (often app/_lib/auth.js), you likely already did this in the previous lecture: 

// app/_lib/auth.js
import NextAuth from "next-auth";
import Google from "next-auth/providers/google";

const authConfig = {
  providers: [
    Google({
      clientId: process.env.AUTH_GOOGLE_ID,
      clientSecret: process.env.AUTH_GOOGLE_SECRET,
    }),
  ],
  // ...callbacks/pages/etc.
};

export const {
  auth,
  handlers: { GET, POST },
  signIn,
  signOut,
} = NextAuth(authConfig);

2) Create the server action for signing out

In your server actions file (example: app/_lib/actions.js), add: 

// app/_lib/actions.js
"use server";

import { signOut } from "./auth";

export async function signOutAction() {
  await signOut({ redirectTo: "/" });
}
What this does ü§ì
  • "use server" forces these exports to execute on the server only.
  • signOut({ redirectTo: "/" }) logs the user out and navigates them back to the homepage.
  • No arguments are needed for this action.

3) Wire the action to your Sign Out button via <form action>

Even if your sign-out UI is inside a client component (like a sidebar), you can still call the server action through a form. 

// components/SignOutButton.jsx (example)
"use client";

import { signOutAction } from "@/app/_lib/actions";

export default function SignOutButton() {
  return (
    <form action={signOutAction}>
      <button className="your-button-classes">
        Sign out
      </button>
    </form>
  );
}
Why this works ‚úÖ

Submitting the form triggers the server action. Even though the button is rendered by a client component, the action itself is executed server-side.

üßØ Common pitfalls and fixes

1) Import path issues

If your import alias @/ is not configured, use a relative import. Ensure the server action is imported from the correct location.

2) Missing "use server"

If you forget "use server" at the top of the actions file, Next.js will not treat the function as a server action.

3) Sidebar is a client component ‚Äî that is fine

The important part is: don‚Äôt try to execute server-only logic directly in the browser. Use the <form action> pattern.
