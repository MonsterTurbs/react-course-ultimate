1) What Are Controlled Elements?

Default behavior: form fields (input, select) maintain their own state inside the DOM. üåê

That means the ‚Äúsource of truth‚Äù for the current value lives in the HTML element itself. In React, we usually prefer to keep state inside the React app (component state), not scattered in the DOM.

üéõÔ∏è

Controlled element (definition)

A form element whose value is controlled by React state. React owns the data; the DOM only displays it.

Why control inputs? ‚úÖ

  • Easier to read values (you already have them in state).
  • Keeps state in one central place (inside React, not inside the DOM).
  • Makes it easy to reset the form after submission.
  • Helps keep UI and state always in sync.

2) The 3-Step Pattern to Control an Input ‚úÖ

  1. Create state (with useState)
  2. Bind state to the element via value={state}
  3. Update state on every change via onChange

Example: control the ‚Äúdescription‚Äù text input ‚úçÔ∏è

import { useState } from "react";

function Form() {
  const [description, setDescription] = useState("");

  return (
    <input
      type="text"
      placeholder="Item..."
      value={description}
      onChange={(e) => setDescription(e.target.value)}
    />
  );
}

Important: if you set value={description} but forget onChange, the input becomes ‚Äúlocked‚Äù. üîí

Because React always forces the value to match state, and state never changes.

3) Understanding e.target.value (Beginner-Friendly)

üß™

What is e?

It‚Äôs the event object React passes to your handler when an event happens (here: change event).

What do these mean?

  • e.target ‚Üí the element that triggered the event (the input/select)
  • e.target.value ‚Üí the current value inside that element
onChange={(e) => {
  // e.target is the <input>
  // e.target.value is what the user typed
  setDescription(e.target.value);
}}

What happens on every keystroke? ‚å®Ô∏è

  1. User types ‚Üí change event fires
  2. setDescription(...) updates state
  3. React re-renders
  4. value={description} updates the input to match state

4) Control the <select> (Quantity) üîΩ

A) Create quantity state

Default should be 1 (the lecture temporarily uses 5 just to show the effect).

const [quantity, setQuantity] = useState(1);

<select value={quantity} onChange={...}>
  ...
</select>

B) Update quantity on change

<select
  value={quantity}
  onChange={(e) => setQuantity(Number(e.target.value))}
>
  ...
</select>

React gives e.target.value as a string, even if your options look numeric. Converting with Number(...) prevents subtle bugs.

Why convert to a number? üî¢

  • Without conversion: quantity becomes "10" (string)
  • With conversion: quantity becomes 10 (number)

DevTools makes it easy to spot: strings show quotes; numbers don‚Äôt.

5) Use the Form Values in handleSubmit üßæ

Controlled elements make it simple: by the time you submit, you already have description and quantity in state.

Create the item object (as in the lecture)

function handleSubmit(e) {
  e.preventDefault();

  // Guard clause: do nothing if empty
  if (!description) return;

  const newItem = {
    id: Date.now(),
    description,
    quantity,
    packed: false,
  };

  console.log(newItem);

  // Reset form back to initial state
  setDescription("");
  setQuantity(1);
}

Guard clause

A simple early return that prevents submitting invalid data (e.g., empty description).

Why resetting is easy now: just set state back to defaults. React updates the form automatically. üôÇ

6) A Key React Idea: Data Cannot Flow Sideways üìå

Problem introduced at the end of the lecture:

You create newItem in Form, but you want it to appear in PackingList. These are sibling components.

Why props alone won‚Äôt solve it

  • Props flow down the component tree.
  • Form is not the parent of PackingList, so it cannot directly pass props to it.
  • To share data between siblings, you typically lift state up to the common parent (often App).
