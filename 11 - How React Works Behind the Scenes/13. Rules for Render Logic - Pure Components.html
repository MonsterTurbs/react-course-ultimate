Rules for Render Logic ‚Äî Pure Components

A print-friendly reviewer explaining render logic vs event handlers, what ‚Äúpure‚Äù means in React, and what you must avoid during rendering.

üß† Goal: make your components predictable so React can safely render, re-render, and reconcile without surprises.

1) Two kinds of logic inside a React component

üß© Render Logic (runs during render)

Runs immediately Render logic is all code executed when the component function runs (i.e., when React renders/re-renders it). It‚Äôs used to describe what the UI should look like.

  • Top-level code inside the component function (variables, derived values, conditional logic)
  • The return JSX
  • Any helper function you call inside JSX (because it runs during render too)
‚úÖ Think: ‚ÄúWhat should the UI snapshot look like right now?‚Äù

üñ±Ô∏è Event Handlers (run after user/system events)

Runs later Event handlers run because something happened (click, input change, submit, hover, etc.). This is where you typically do things.

  • State updates (e.g., setState)
  • HTTP requests / API calls
  • Reading user input
  • Navigation
  • Starting timers
‚úÖ Think: ‚ÄúAn action happened‚Äîupdate something.‚Äù
Key idea: React requires components to be pure in render logic. Event handlers can (and often should) cause side effects.

2) Side effects and pure functions (quick refresher)

üåç What is a side effect?

A side effect happens when a function:

  • depends on data outside its scope (like current time, global variables), and/or
  • modifies something outside its scope (mutating objects, DOM updates, network requests, timers).

A good mental model: a side effect is a function ‚Äúinteracting with the outside world.‚Äù

üßº What is a pure function?

A pure function has no side effects and is predictable: same input ‚Üí same output. 

// ‚úÖ Pure: predictable
function circleArea(r) {
  return Math.PI * r * r;
}

// ‚ùå Impure: output changes even with the same input
function nowLabel() {
  return `Time: ${new Date().toISOString()}`;
}

// ‚ùå Impure: mutates outside state
let counter = 0;
function inc() {
  counter = counter + 1;
}

Side effects are not ‚Äúbad‚Äù‚Äîapps need them. The important part is where you do them in React.

3) The big rule: render logic must be pure

Rule: If a component instance receives the same props (same input), it should return the same JSX output. That means: no side effects inside render logic.

‚úÖ Allowed in render logic

  • ‚úÖ Deriving UI values from props/state (e.g., sorting, filtering, formatting)
  • ‚úÖ Conditional rendering (ternaries, &&)
  • ‚úÖ Mapping arrays to JSX
  • ‚úÖ Calling helper functions that are also pure

‚õî Not allowed in render logic (common side effects)

  • No Network requests (fetch/axios) inside the component body
  • No Timers (setTimeout, setInterval) created during render
  • No Direct DOM operations (document.querySelector, addEventListener, manual DOM writes)
  • No Mutating variables/objects outside the component scope
  • No Mutating props (this is why ‚Äúprops are read-only‚Äù is a hard rule)
  • No Updating state or refs during render

Why ‚Äúno state updates in render‚Äù is critical: it can cause an infinite render loop.

// ‚ùå Never do this
function Bad() {
  const [count, setCount] = React.useState(0);
  setCount(count + 1); // triggers render ‚Üí triggers setState ‚Üí ... forever
  return <div>{count}</div>;
}

‚ö†Ô∏è ‚ÄúBut I use console.log() and Math.random() in render‚Ä¶‚Äù

These are technically side effects / non-deterministic operations, but they‚Äôre usually harmless for learning/debugging. Still, use them intentionally:

  • üßæ console.log: fine for debugging (remove/limit in production).
  • üé≤ Math.random / new Date(): makes output unpredictable; can cause surprising UI changes across renders.

4) Where do side effects go, then?

Option A: Event handlers ‚úÖ

If the effect should happen because the user did something (click, submit, type), do it in an event handler. 

function SearchBox() {
  const [query, setQuery] = React.useState("");

  async function handleSubmit(e) {
    e.preventDefault();
    // ‚úÖ Side effect allowed here
    const res = await fetch(`/api/search?q=${encodeURIComponent(query)}`);
    const data = await res.json();
    console.log(data);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <button>Search</button>
    </form>
  );
}

Option B: useEffect ‚úÖ (run effect after render)

If you need an effect to run when the component mounts or when some value changes, register it with useEffect. 

function Profile({ userId }) {
  const [user, setUser] = React.useState(null);

  React.useEffect(() => {
    let isMounted = true;

    async function load() {
      const res = await fetch(`/api/users/${userId}`);
      const data = await res.json();
      if (isMounted) setUser(data);
    }

    load();
    return () => { isMounted = false; };
  }, [userId]);

  return <div>{user ? user.name : "Loading..."}</div>;
}

üìå Important: useEffect runs after React renders (and after commit), so it doesn‚Äôt break render purity.

5) Quick self-check (interview-friendly)

‚úÖ If someone asks: ‚ÄúWhat is render logic?‚Äù

  • Render logic is everything that runs when the component function runs and helps produce JSX.
  • It must be pure: same props/state ‚Üí same JSX output.

‚úÖ If someone asks: ‚ÄúWhy can‚Äôt we mutate props?‚Äù

  • Mutating props is a side effect (it changes data outside the component scope).
  • It breaks predictability across renders and can confuse reconciliation.

‚úÖ If someone asks: ‚ÄúWhere do side effects belong?‚Äù

  • Event handlers (user-triggered effects)
  • useEffect for effects that must run after render
