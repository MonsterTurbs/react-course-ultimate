üî§ Step 2: Add a custom Google Font (Roboto Mono) and register it in Tailwind

2A) Include the font in your index.html

In the lecture, the font comes from Google Fonts (Roboto Mono) with multiple weights (300‚Äì700). You typically paste a <link> in your HTML head. 

<!-- index.html (inside <head>) -->

2B) Define a Tailwind font family

In tailwind.config.js, you can define a font stack. In the lecture, a custom name like pizza is used. 

// tailwind.config.js
export default {
  theme: {
    fontFamily: {
      pizza: ['Roboto Mono', 'monospace'],
    },
  },
};

2C) Use it in JSX/HTML with Tailwind classes

<header className="font-pizza">
  ...
</header>

‚úÖ What you should notice

Once Tailwind knows a font name (like pizza), you can apply it using font-pizza anywhere.

üåç Make the font apply everywhere (override the default sans)

Using font-pizza manually is fine, but usually you want one font for the whole app. Tailwind uses font-sans as a default ‚Äúbase‚Äù in many setups. If you override the sans family, the whole UI updates. 

// tailwind.config.js

  theme: {
    fontFamily: {
      sans: ['Roboto Mono', 'monospace'],
      // You can keep other families too (serif/mono) if you want
    },
  },
};

‚ö†Ô∏è Common confusion

If you replace theme.fontFamily entirely, you may remove other defaults. If you still want defaults plus your changes, use extend (next section).

üß± Core Concept: theme overrides, theme.extend adds

This is one of the most important Tailwind configuration rules:

  • ‚úÖ theme: { ... } ‚Äî replaces Tailwind‚Äôs defaults for that category
  • ‚úÖ theme: { extend: { ... } } ‚Äî keeps defaults and adds your custom tokens

üö´ If you override the wrong way, classes can disappear

Example: if you override fontSize without using extend, then classes like text-sm may stop existing.

Example: Add a custom color called pizza (the correct ‚Äúextend‚Äù way)

// tailwind.config.js
export default {
  theme: {
    extend: {
      colors: {
        pizza: '#123456',
      },
    },
  },
};

Then you can use it like bg-pizza / text-pizza in your markup.

Example: Add a custom font size huge

// tailwind.config.js
export default {
  theme: {
    extend: {
      fontSize: {
        huge: '80rem',
      },
    },
  },
};

<h1 className="text-huge">Big text</h1>

üì± Step 3: Fix mobile height issues by redefining screen to use 100dvh

The lecture calls out a real mobile browser issue: 100vh can be unreliable because mobile UI (address bar, toolbars) changes the visible viewport. A modern fix is dvh (dynamic viewport height).

‚úÖ Why dvh helps

100dvh tracks the actual visible viewport more accurately on mobile, reducing layout jumps and ‚Äúcut off‚Äù sections.

Override Tailwind‚Äôs h-screen behavior

In Tailwind, h-screen maps to the theme‚Äôs height.screen. You can redefine it: 

// tailwind.config.js
export default {
  theme: {
    extend: {
      height: {
        screen: '100dvh',
      },
    },
  },
};

After this, hovering your h-screen class in VS Code should show height: 100dvh.

‚ö†Ô∏è Real-world reminder

Some older browsers may not fully support newer viewport units. For modern React apps, dvh is increasingly safe and recommended.
