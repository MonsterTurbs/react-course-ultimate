üéØ Goal and UI behavior

We want to add ‚Äúmovie selection‚Äù behavior: when a user clicks a movie on the left list, the app shows details on the right. Then the user can go back (close details) with a back button.

Concept: More dynamics = more state. üôÇ

Selection happens on the left (movie list), but details display on the right. Because multiple parts of the UI need the same information, the state should live in the parent.

üß© New state: selectedId

Add state in the App component so both panels can access it. Start with null (meaning: ‚Äúno movie selected‚Äù). 

// App.jsx
const [selectedId, setSelectedId] = useState(null);
  • ‚úÖ null = nothing selected
  • ‚úÖ when a user clicks a movie, set it to that movie‚Äôs imdbID
  • ‚úÖ the right panel can render details based on this ID

üÜî Why store the ID (not the full movie object)?

The search results from OMDb are limited (title, year, poster, imdbID). The details view needs more data, which requires another API call.

Practical mindset: Store the minimum identifier you need (ID), then fetch full details when required. üí°
  • Search list gives { Title, Year, Poster, imdbID, Type }
  • Details view will fetch ‚Äúfull movie details‚Äù using imdbID
  • This keeps state simple, avoids duplicating partial + full objects

üñ±Ô∏è Selecting a movie (click handler)

The state ‚Äúlives‚Äù in App, but the click happens inside Movie list items. So we: (1) create a handler in App, (2) pass it down to MovieList, (3) pass it down to Movie, (4) call it on click.

1) Handler in App

function handleSelectMovie(id) {
  setSelectedId(id);
}

2) Pass down to MovieList

<MovieList movies={movies} onSelectMovie={handleSelectMovie} />

3) MovieList forwards to Movie

function MovieList({ movies, onSelectMovie }) {
  return (
    <ul className="list list-movies">
      {movies.map(movie => (
        <Movie
          key={movie.imdbID}
          movie={movie}
          onSelectMovie={onSelectMovie}
        />
      ))}
    </ul>
  );
}

4) Movie calls it on click

function Movie({ movie, onSelectMovie }) {
  return (
    <li onClick={() => onSelectMovie(movie.imdbID)}>
      <img src={movie.Poster} alt={`${movie.Title} poster`} />
      <h3>{movie.Title}</h3>
      <p>
        <span>üóìÔ∏è</span> {movie.Year}
      </p>
    </li>
  );
}
Pattern: ‚ÄúState up, actions down.‚Äù üôÇ Parent owns the state; children receive callbacks to request updates.

‚Ü©Ô∏è Closing the details

Closing simply means setting selectedId back to null. 

function handleCloseMovie() {
  setSelectedId(null);
}

Then pass it into MovieDetails and attach it to the back button. No wrapper function needed because we are not passing arguments. 

<button className="btn-back" onClick={onCloseMovie}>
  &larr;
</button>

üîÅ Toggle behavior: click the same movie closes

Nice UX improvement: if the user clicks the already-selected movie again, we close the details instead of re-selecting.

This is a perfect situation for the state setter callback because we are computing the next state from the current state. 

function handleSelectMovie(id) {
  setSelectedId(currentSelectedId =>
    id === currentSelectedId ? null : id
  );
}
Rule of thumb: If next state depends on current state, use the callback form. ‚úÖ

‚úÖ Key takeaways and common mistakes

Key takeaways

  • üéØ More UI behavior = more state (choose state location wisely).
  • üè† Shared state should live in the closest common parent.
  • üÜî Store identifiers when you need to fetch more data later.
  • üß∑ Prop drilling is fine when it‚Äôs only 1 level.
  • üîÅ Use callback setters when next state depends on previous state.

Common mistakes

  • ‚ö†Ô∏è Putting selectedId in the wrong component (then the other panel can‚Äôt access it).
  • ‚ö†Ô∏è Returning two sibling elements from a ternary without wrapping (use a fragment).
  • ‚ö†Ô∏è Calling a handler immediately: onClick={onSelectMovie(movie.imdbID)} (this runs during render).
  • ‚úÖ Correct: onClick={() => onSelectMovie(movie.imdbID)}
