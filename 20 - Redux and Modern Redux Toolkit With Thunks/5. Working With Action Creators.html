1) Big idea: What is an action creator?

üß© An action creator is just a function that returns an action object. It is a convention (not required by Redux), but it is widely used because it keeps your action shapes consistent and reduces bugs.

‚úÖ Why teams like action creators
  • Less manual typing of type strings (fewer typos).
  • Reusable: call deposit(500) anywhere instead of rewriting an object.
  • Single place to change payload shape later (e.g., adding fields).
‚ö†Ô∏è Reminder

Redux works perfectly fine without action creators. They just make code cleaner and safer.

2) Action object shape (quick refresher)

An action is usually an object like this: 

// Typical action
{
  type: "account/deposit",
  payload: 500
}

üîç type describes what happened, and payload carries the data needed. (Payload can be a number, string, or another object.)

3) Build action creators (one per action)

In the bank-account example, we create one action creator for each possible action: deposit, withdraw, requestLoan, and payLoan. 

// store.js (or actions.js)

// 1) Deposit
export function deposit(amount) {
  return {
    type: "account/deposit",
    payload: amount,
  };
}

// 2) Withdraw
export function withdraw(amount) {
  return {
    type: "account/withdraw",
    payload: amount,
  };
}

// 3) Request loan (payload is an OBJECT)
export function requestLoan(amount, purpose) {
  return {
    type: "account/requestLoan",
    payload: { amount, purpose },
  };
}

// 4) Pay loan (no payload needed)
export function payLoan() {
  return {
    type: "account/payLoan",
  };
}
üß† Key detail

Payload can be anything. If you need multiple values (like loan amount + purpose), use an object payload.

4) Use them with store.dispatch

Instead of manually dispatching objects, you dispatch the result of calling an action creator. 

// BEFORE (manual action object)
store.dispatch({ type: "account/deposit", payload: 500 });

// AFTER (action creator)
store.dispatch(deposit(500));

‚úÖ Examples

// Deposit + withdraw
store.dispatch(deposit(500));
store.dispatch(withdraw(200));

console.log(store.getState());

üí≥ Loan examples

// Request a loan (object payload)
store.dispatch(requestLoan(1000, "Buy a cheap car"));
console.log(store.getState());

// Pay the loan
store.dispatch(payLoan());
console.log(store.getState());
‚úÖ Benefit in real projects

If action types get renamed or payload structure changes, you update the action creator once, not every dispatch call in the codebase.

5) Common mistake: typos still happen (but fewer)

Even with action creators, you can still introduce bugs if you type the string wrong. Example: "account/payLoan" vs "account/payloan".

ü™≤ Bug example
// Typo in action type (reducer won't match this case)
export function payLoan() {
  return { type: "account/payloan" }; // ‚ùå wrong capitalization
}

Result: your reducer hits the default case and returns the same state, so it looks like ‚Äúnothing happened.‚Äù

üîé Debug tip
  • Log the dispatched action: console.log(action)
  • Confirm reducer case labels exactly match action.type.
