1) Why Lifting State Up Can Feel Counterintuitive

Sometimes you start building a component, create state there, and it feels natural. Then React ‚Äúforces‚Äù you to move that state into a parent component‚Ä¶ even though the parent doesn‚Äôt directly use it. That can feel weird at first.

The reason is simple: React has one-way data flow ‚¨áÔ∏è. Parents can pass data down to children via props, but siblings can‚Äôt directly share state.

Mental model

If two components need the same data, the ‚Äúowner‚Äù of that data should be above both of them in the component tree.

3) The Problem: State Created in One Sibling, Needed by Another ‚ùå

If you start by building Promotions, you might create local state there: 

function Promotions() {
  const [coupons, setCoupons] = useState([]);
  // ... UI to add coupons
}

Then you build Total and realize it also needs coupons. But because Promotions and Total are siblings, you cannot pass coupons directly from Promotions to Total.

Key rule: Props only flow down from parent to child. There is no ‚Äúprops sideways‚Äù between siblings. ‚ùå

4) The Solution: Lift State Up to the Closest Common Parent ‚¨ÜÔ∏è

Lifting state up means moving state into a component that is a parent of all components that need that state.

In this example, the closest common parent of Promotions and Total is Checkout. So we move coupons state into Checkout. 

Before (problem):
Promotions owns coupons ‚ùå  Total needs coupons

After (solution):
Checkout owns coupons ‚úÖ
  ‚îú‚îÄ Promotions receives coupons + a way to update them
  ‚îî‚îÄ Total receives coupons to compute totals

Checkout (state owner)

function Checkout() {
  const [coupons, setCoupons] = useState([]);

  return (
    <>
      <Promotions coupons={coupons} setCoupons={setCoupons} />
      <Total coupons={coupons} />
    </>
  );
}

Now both components can access the same state because it is passed down from the parent.

5) But‚Ä¶ How Can a Child Update Parent State? (Inverse Data Flow) üîÅ‚¨ÜÔ∏è

After lifting state up, Promotions no longer owns the state. It only receives coupons via props.

  • Props are read-only üß∑ (child cannot mutate them)
  • So the child needs a way to request an update

Solution: pass a setter or handler function down as a prop. Child calls it when user interacts.

Option A: Pass the setter directly (setCoupons)

function Promotions({ coupons, setCoupons }) {
  function handleApply(code) {
    const newCoupon = { id: Date.now(), code };
    setCoupons((coupons) => [...coupons, newCoupon]);
  }

  return (
    <div>
      {/* input + button; on click call handleApply */}
    </div>
  );
}

This is valid, but it couples the child to the exact state shape/update logic.

Option B (common): Pass a handler like onAddCoupon ‚úÖ

function Checkout() {
  const [coupons, setCoupons] = useState([]);

  function handleAddCoupon(code) {
    const newCoupon = { id: Date.now(), code };
    setCoupons((coupons) => [...coupons, newCoupon]);
  }

  return (
    <>
      <Promotions coupons={coupons} onAddCoupon={handleAddCoupon} />
      <Total coupons={coupons} />
    </>
  );
}

function Promotions({ onAddCoupon }) {
  function handleApply(code) {
    onAddCoupon(code); // ‚úÖ child requests update
  }
}

This matches what you did in the Far Away app: you didn‚Äôt pass setItems directly‚Äî you passed a function that uses it.

This pattern is called: child-to-parent communication or inverse data flow. Not because data literally flows up, but because children can trigger updates to parent-owned state.

6) Why This Pattern Is So Important üîë

  • ‚úÖ Lets you share state between siblings without breaking one-way data flow
  • ‚úÖ Keeps a single source of truth for shared data
  • ‚úÖ Makes UI predictable: parent owns state ‚Üí children receive a snapshot via props
  • ‚úÖ Scales naturally as component trees grow
üìå

Rule of thumb

If multiple components must stay in sync, state should live in their closest common parent.
