üß† Big picture: SSG vs SSR in the Pages Router

üèóÔ∏è SSG (Static Site Generation) ‚Äî with getStaticProps

  • Runs at build time (not per request).
  • Resulting HTML is pre-generated and reused for all visitors.
  • Best when data changes rarely.

üåê SSR (Server-Side Rendering) ‚Äî with getServerSideProps

  • Runs at request time (every page request).
  • Ideal when data changes frequently or depends on user/request.
  • Great for dynamic pages that must always be fresh.
‚ö†Ô∏è Dynamic route note: if a route includes a dynamic segment like /cabins/[cabinId], you typically render it dynamically (SSR) unless you provide all possible IDs using getStaticPaths.

üõ£Ô∏è Dynamic routes (Pages Router) refresher

In the Pages Router, a dynamic route is created by naming the file with square brackets:

// pages/cabins/[cabinId].js

Example URLs that hit this page:

  • /cabins/88
  • /cabins/202
  • /cabins/999

üß© Implementing SSR with getServerSideProps

The special feature of getServerSideProps is that it runs on the server and receives a context object that includes route params. That means you can access params.cabinId for /cabins/[cabinId].

‚úÖ Minimal SSR pattern

export async function getServerSideProps({ params }) {
  // 1) Read the dynamic segment
  const cabinId = params.cabinId;

  // 2) Fetch data on the server (request-time)
  const cabin = await getCabin(cabinId);

  // 3) Pass to the page component as props
  return {
    props: { cabin },
  };
}

export default function CabinPage({ cabin }) {
  return (
    <div style={{ maxWidth: 900, margin: "0 auto", marginTop: 24 }}>
      <CabinView cabin={cabin} />
    </div>
  );
}
‚úÖ What you get: Each time a user requests /cabins/88, Next.js runs getServerSideProps, fetches the latest cabin data, and renders the page HTML on the server.

üîé Where does params come from?

Next.js builds it from your file name [cabinId].js. Whatever appears in the URL at that spot becomes: params.cabinId.

üß† Memory tip: File name controls param name. If the file is [productId].js, you read params.productId.

üÜö getStaticProps vs getServerSideProps (clear comparison)

üèóÔ∏è getStaticProps (SSG)

  • Runs at build time
  • Fast for users, highly cacheable
  • Best for ‚Äúcontent-like‚Äù pages (rarely changing)
export async function getStaticProps() {
  const cabins = await getCabins();
  return { props: { cabins } };
}

üåê getServerSideProps (SSR)

  • Runs on every request
  • Fresh data each time
  • Best for ‚Äúlive‚Äù pages (frequent updates / user-specific)
export async function getServerSideProps({ params }) {
  const cabin = await getCabin(params.cabinId);
  return { props: { cabin } };
}
üß≠ Rule of thumb:
  • If data changes rarely ‚Üí prefer SSG.
  • If data changes often or depends on request/user ‚Üí prefer SSR.

üß† Can dynamic routes be SSG?

Yes. But Next.js needs to know all possible dynamic values at build time. In the Pages Router, you do that with getStaticPaths (similar idea to generateStaticParams in the App Router).

üß™ Conceptual SSG setup for a dynamic route

export async function getStaticPaths() {
  // Tell Next.js which IDs exist
  const cabins = await getCabins();

  return {
    paths: cabins.map((c) => ({ params: { cabinId: String(c.id) } })),
    fallback: false,
  };
}

export async function getStaticProps({ params }) {
  const cabin = await getCabin(params.cabinId);
  return { props: { cabin } };
}
‚ö†Ô∏è If you don‚Äôt provide paths, Next.js can‚Äôt prebuild unknown IDs. That‚Äôs why SSR is the simpler default for many dynamic pages.

‚è±Ô∏è ISR (Incremental Static Regeneration) in Pages Router

ISR is a middle-ground between SSG and SSR. You generate a page statically, but Next.js will re-generate it in the background after some time.

‚úÖ How ISR is expressed in getStaticProps

export async function getStaticProps() {
  const cabins = await getCabins();

  return {
    props: { cabins },
    revalidate: 3600, // seconds (e.g., once per hour)
  };
}
üß† ISR behavior:
  • Users still get fast static HTML.
  • After revalidate seconds, Next.js refreshes the page in the background.
  • The next request after regeneration receives updated content.

‚úÖ Practical checklist (avoid common mistakes)

  1. Use getServerSideProps when you need fresh data per request.
  2. Read dynamic params via { params } and match the file name (e.g., [cabinId].js ‚Üí params.cabinId).
  3. Return props using return { props: { ... } }.
  4. Make sure your page component accepts the same prop names you returned.
  5. If you want SSG for dynamic routes, add getStaticPaths + getStaticProps.
‚ö†Ô∏è Remember: You can also fetch on the client (useEffect / React Query / SWR), but Next.js exists mainly to help with server rendering for faster first load.
