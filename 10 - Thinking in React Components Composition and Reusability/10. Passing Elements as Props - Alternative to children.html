üí° Key idea

‚ÄúComponent composition‚Äù means: build flexible components by letting the caller provide what should be rendered inside. Most of the time, React uses children for this.

‚úÖ Why this is nice
  • Clean usage: the JSX you pass looks like real content inside <Box>...</Box>.
  • Flexible: you can pass one element, many elements, or fragments.
  • Common React convention: most libraries and teams expect this pattern.

üß© Alternative: Passing JSX via an explicit prop (e.g., element)

Instead of receiving children, a component can receive a prop that contains JSX. In the lecture, the example prop name was element, but it can be anything.

  • Now: JSX is passed explicitly via a named prop ‚Üí read through element.

üß™ Under the hood

In practice, React can render either approach the same way because both ultimately pass React elements into a component. The choice is more about readability, conventions, and flexibility.

‚úÖ When is the explicit-prop pattern actually useful?

Most of the time, children is preferred. But passing JSX via explicit props becomes useful when you need multiple named ‚Äúslots‚Äù.

‚úÖ Good use cases
  • Named slots: header/body/footer (or left/right panels)
  • APIs like React Router: patterns that accept element/Component props
  • Clarity: you want the caller to know exactly what each slot is for
‚ö†Ô∏è Trade-offs
  • Usage can look noisier: element={...} everywhere
  • More props to manage if you only need one slot
  • Team convention: most React codebases default to children
