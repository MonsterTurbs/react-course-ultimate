1) What you are building

You want to block access to the ‚Äúguest area‚Äù routes (e.g., /account) when the user is not logged in. If they‚Äôre unauthorized, they should be redirected to a sign-in page.

üß† Key idea

Middleware runs on every request before the route renders (unless you limit it with a matcher). This is perfect for authorization checks.

2) Middleware file location and conventions

  • File name: middleware.js
  • Location: project root (same level as package.json), not inside app/
  • Export: a function named middleware (or export a compatible middleware function)

3.2 Redirecting with NextResponse

You can force a redirect using NextResponse.redirect(). But you must avoid redirect loops. 

// middleware.js (root)
import { NextResponse } from "next/server";

export function middleware(request) {
  // Always redirect to /about (demo only)
  return NextResponse.redirect(new URL("/about", request.url));
}

‚ö†Ô∏è Common problem: ‚ÄúToo many redirects‚Äù

Middleware runs for every route by default. If you redirect to /about and middleware still runs on /about, you get an infinite loop.

3.3 Fixing loops with a matcher

Use export const config and a matcher to restrict which routes execute the middleware. 

// middleware.js (root)
import { NextResponse } from "next/server";

export function middleware(request) {
  return NextResponse.redirect(new URL("/about", request.url));
}

export const config = {
  matcher: ["/account"], // middleware runs ONLY on /account
};

4) Real goal: Protect /account with NextAuth/Auth.js

NextAuth (Auth.js) provides an auth helper that can act as:

  • ‚úÖ a function to read the current session in server components
  • ‚úÖ a middleware-compatible function to protect routes

4.1 Use Auth.js middleware in middleware.js

// middleware.js (root)

export const middleware = auth;

export const config = {
  matcher: ["/account"],
};

‚úÖ What this does

Now, whenever a request targets /account, Auth.js middleware runs and can decide whether the user is allowed.

5) Required Auth.js config: the authorized callback

Middleware alone isn‚Äôt enough ‚Äî you also configure a callback in your Auth.js setup. This callback returns true if the request is authorized, otherwise false. 

// app/_lib/auth.js (or your auth config file)
import NextAuth from "next-auth";
import Google from "next-auth/providers/google";

const config = {
  providers: [
    Google({
      clientId: process.env.AUTH_GOOGLE_ID,
      clientSecret: process.env.AUTH_GOOGLE_SECRET,
    }),
  ],

  callbacks: {
    authorized({ auth, request }) {
      // ‚úÖ simplest rule: allow only logged-in users
      return !!auth?.user;
    },
  },
};

export const { auth, handlers: { GET, POST } } = NextAuth(config);

üß© Explain it like I‚Äôm new

  • auth here means ‚Äúcurrent authentication/session info‚Äù
  • auth?.user exists when the user is logged in
  • !! converts a value into a strict boolean (true/false)
  • If you return false, Auth.js will redirect to a sign-in route

  1. üö∂ User tries to visit /account
  2. üõë middleware.js runs (because matcher matches /account)
  3. üîé Auth.js executes callbacks.authorized(...)
  4. ‚úÖ If logged in ‚Üí allow route rendering
    ‚ùå If not logged in ‚Üí redirect to a sign-in page

7) Practical checklist (debug fast)

‚úÖ File placement

  • middleware.js is in the project root (not app/)
  • Matcher includes the route(s) you want to protect, e.g. "/account"

‚úÖ Auth.js wiring

  • You export auth from your Auth.js setup file
  • callbacks.authorized returns boolean
  • Use optional chaining: !!auth?.user

‚ö†Ô∏è Avoid common traps

  • Redirect loops if middleware matches the redirect target
  • Protect only what you need (don‚Äôt over-match everything)
  • Remember: middleware runs per-request, so keep it fast
