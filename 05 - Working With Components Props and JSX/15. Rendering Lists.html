üß† What ‚Äúrendering a list‚Äù means

Rendering a list means: you have an array (usually data) and you want to create one component per item.

Example scenario üçï

You have an array of pizza objects. For every pizza object, render one <Pizza /> component.

Goal ‚úÖ

If the array has 4 items ‚Üí render 4 components. If it has 10 items ‚Üí render 10 components.

üîÅ The core tool: JavaScript .map()

React doesn‚Äôt give you a special ‚Äúlist component.‚Äù Instead, you use the JavaScript you already know. The key method is Array.prototype.map().

Why map works in JSX ‚úÖ

map returns a new array. In React, an array of JSX elements can be rendered directly.

Basic pattern (one component per item) üß©

function Menu() {
  return (
    <div>
      {pizzaData.map((pizza) => (
        <Pizza />
      ))}
    </div>
  );
}

In real usage, you pass props so each component gets its own data (next section).

üì¶ Passing props: per-field vs whole object

When you map, you usually want each rendered component to receive data. There are two common approaches:

Approach A: pass individual fields üßæ

Works, but can get repetitive when there are many fields.

{pizzaData.map((pizza) => (
  <Pizza
    name={pizza.name}
    ingredients={pizza.ingredients}
    price={pizza.price}
    photoName={pizza.photoName}
  />
))}

Approach B: pass the whole object üß†

Common in real projects: fewer props in the parent; child reads what it needs.

{pizzaData.map((pizza) => (
  <Pizza pizzaObj={pizza} />
))}

Then inside Pizza:

function Pizza(props) {
  const p = props.pizzaObj;

  return (
    <li className="pizza">
      <img src={p.photoName} alt={p.name} />
      <div>
        <h3>{p.name}</h3>
        <p>{p.ingredients}</p>
        <span>{p.price}</span>
      </div>
    </li>
  );
}

Practical tip ‚úÖ

Passing the full object is often cleaner, especially when the child needs multiple properties.

üîë Keys: why React warns you

When you render a list, React will warn: ‚ÄúEach child in a list should have a unique key prop.‚Äù

What you must do ‚ö†Ô∏è

Add a key prop to the top-level element you return from map (usually the component).

Example: using a unique property ‚úÖ

{pizzaData.map((pizza) => (
  <Pizza key={pizza.name} pizzaObj={pizza} />
))}

In the lecture, name is unique, so it works as a key. (In real apps, an id is usually the best key.)

Why keys exist (simple explanation) üß†

React uses keys to track items between renders, enabling performance optimizations and correct UI updates when lists change.

üß± Semantic markup: ul + li

If you are rendering a list of items, use semantic HTML: <ul> (unordered list) and <li> (list item).

Recommended structure ‚úÖ

function Menu() {
  return (
    <main className="menu">
      <h2>Our menu üçï</h2>

      <ul className="pizzas">
        {pizzaData.map((pizza) => (
          <Pizza key={pizza.name} pizzaObj={pizza} />
        ))}
      </ul>
    </main>
  );
}

üö´ Why forEach doesn‚Äôt work in JSX

It feels natural to say ‚Äúrender one pizza for each pizza,‚Äù but forEach is not the right tool.

forEach ‚ùå

forEach returns undefined, so React has nothing to render.

// ‚ùå This produces nothing useful to render
<ul>
  {pizzaData.forEach((pizza) => (
    <Pizza pizzaObj={pizza} />
  ))}
</ul>

map ‚úÖ

map returns a new array of JSX, which React can render.

// ‚úÖ Returns an array of <Pizza /> elements
<ul>
  {pizzaData.map((pizza) => (
    <Pizza key={pizza.name} pizzaObj={pizza} />
  ))}
</ul>

Memorize this sentence ‚úÖ

In JSX, you must return a value. map returns an array (renderable). forEach returns nothing (not renderable).

üìù Quick review + common mistakes
