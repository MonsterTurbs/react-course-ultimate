üß≠ Big picture: what we are refactoring

In Atomic Blog, the app generates 30 random posts on initial load, then filters them using a search query. It also supports:

  • ‚ûï Adding a new post
  • üßπ Clearing all posts
  • üîé Filtering posts by what the user types (e.g., ‚Äúfire‚Äù)

‚ö†Ô∏è Why this matters

The app currently suffers from prop drilling: components that don‚Äôt need data still receive it just to pass it further down. This becomes noisy, error-prone, and harder to maintain.

üß© Example of prop drilling (concept)

Imagine data starting in App but needed in a deep component like List. Without context, you often pass props like:

  • App ‚Üí Main ‚Üí Posts ‚Üí List
  • Some middle components are just ‚Äúpass-through‚Äù components.

üß† Quick Context API recap (3 parts)

1) Provider üß±

A special component that broadcasts a value to its descendants.

2) Value üéÅ

The data you want to share (usually state + handlers), passed via the value prop.

3) Consumers üëÄ

Components that read the shared value (typically using useContext).

‚úÖ Key mental model

Context lets you move from ‚Äúpass props through the tree‚Äù to ‚Äúread shared data where you need it‚Äù.

1Ô∏è‚É£ Step 1 ‚Äî Create a context (createContext)

Start by importing createContext from React, then create a context instance. In this lecture, the context is named PostContext because it will share post-related data.

‚ö†Ô∏è About ‚Äúdefault values‚Äù

You can pass a default value to createContext(defaultValue), but in practice many projects use null (or nothing) because the provider will supply the real value. A default value cannot ‚Äúevolve‚Äù on its own; it‚Äôs not a replacement for state. 

import { createContext } from "react";

// Usually exported so other files can consume it
export const PostContext = createContext(null);

üß† Why the name starts with uppercase

You‚Äôll soon use PostContext.Provider inside JSX. JSX ‚Äúcomponents‚Äù are conventionally PascalCase, so we name contexts similarly.

2Ô∏è‚É£ Step 2 ‚Äî Wrap your JSX with the Provider

Every context has a Provider component. Wrap the portion of the tree that needs access. Commonly, you wrap the entire application subtree. 

function App() {
  return (
    <PostContext.Provider>
      {/* Everything inside can read PostContext */}
      <Header />
      <Main />
      <Footer />
    </PostContext.Provider>
  );
}

üß© What changes at this step?

Nothing visible yet. Wrapping with a Provider alone does not share anything until you pass a value.

3Ô∏è‚É£ Step 3 ‚Äî Provide a value object (state + handlers)

The value is passed via the value prop. A common pattern is an object that groups everything consumers need.

üéÅ What goes into the value?

  • üóÇÔ∏è Derived data: searchedPosts
  • üßπ Event handlers: handleAddPost, handleClearPosts
  • üîé UI state: searchQuery and setSearchQuery
function App() {
  // Example: derived state + state + handlers (names based on lecture)
  const searchedPosts = /* filter posts based on searchQuery */;

  function handleAddPost(newPost) {
    // add logic
  }

  function handleClearPosts() {
    // clear logic
  }

  return (
    <PostContext.Provider
      value={{
        posts: searchedPosts,
        onAddPost: handleAddPost,
        onClearPosts: handleClearPosts,
        searchQuery,
        setSearchQuery,

      <Header />
      <Main />
      <Footer />
    </PostContext.Provider>

}

‚úÖ What you gained immediately

You now have a single ‚Äúshared source‚Äù for these values. Next, you will stop passing them through props and instead read them directly where needed.

‚ö†Ô∏è About ‚Äúone context per domain‚Äù

In real projects, it‚Äôs common to create one context for a single ‚Äúdomain‚Äù (e.g., posts) and another for a separate domain (e.g., search). The lecture keeps them together for learning simplicity.

üõ†Ô∏è Common pitfalls & best practices

‚ùå Pitfall: Provider without value

If you forget value, consumers may receive null (or an unusable default), causing runtime errors.

‚ö†Ô∏è Pitfall: Over-sharing

Putting too many unrelated values into one context can cause unnecessary re-renders and makes the context harder to reason about. Consider splitting contexts by feature/domain.

üß† Naming conventions

  • Context name: PostContext
  • Provider component: PostContext.Provider
  • Value shape: keep it predictable (e.g., posts, onAddPost)

üß© Practical tip

Think of context value as your ‚Äúpublic API‚Äù for the feature. Only expose what consumers truly need.

üß™ Mini example: a clean template you can reuse

This is a typical structure you‚Äôll see in many React codebases: create context, wrap provider, pass state + actions. 

import { createContext, useMemo, useState } from "react";

export const PostContext = createContext(null);

export default function App() {
  const [searchQuery, setSearchQuery] = useState("");
  const [posts, setPosts] = useState([]);

  const searchedPosts = posts.filter(
    (p) => p.title.includes(searchQuery) || p.body.includes(searchQuery)
  );

  function handleAddPost(newPost) {
    setPosts((cur) => [newPost, ...cur]);
  }

  function handleClearPosts() {
    setPosts([]);
  }

  // Optional: memoize the value to reduce re-renders in consumers
  const value = useMemo(
    () => ({
      posts: searchedPosts,
      onAddPost: handleAddPost,
      onClearPosts: handleClearPosts,
      searchQuery,
      setSearchQuery,
    }),
    [searchedPosts, searchQuery]
  );

  return (
    <PostContext.Provider value={value}>
      <Header />
      <Main />
    </PostContext.Provider>
  );
}

‚ö†Ô∏è Note about useMemo

Memoization is optional and depends on your needs. The lecture focuses on understanding context flow first.
