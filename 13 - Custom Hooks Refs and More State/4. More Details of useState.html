1) Core Reminder: useState(initialValue) uses the initial value only once

üí° Rule Initial state is read only on the initial render (mount)
React looks at the value you pass to useState(...) only when the component first mounts. On later re-renders, React ignores that ‚Äúinitial value‚Äù and uses the stored state instead.

Example scenario: ‚ÄúisTop‚Äù depends on IMDB rating

You might want a piece of state called isTop which is true when imdbRating > 8.

‚ùå Common mistake

// ‚ùå Looks reasonable, but can fail in async data scenarios
const [isTop, setIsTop] = useState(Number(imdbRating) > 8);
ü§î Why this fails
  • When the component mounts, the movie data often hasn‚Äôt arrived yet.
  • So imdbRating is initially undefined.
  • Number(undefined) > 8 becomes false.
  • That false becomes the initial state, and it stays that way until you explicitly update it.

2) Fix Option A: Update the state when the rating changes (using useEffect)

If you truly need isTop as state (usually you don‚Äôt), you can sync it when imdbRating changes. 

const [isTop, setIsTop] = useState(false);

useEffect(() => {
  if (!imdbRating) return;                 // prevent undefined issues
  setIsTop(Number(imdbRating) > 8);
}, [imdbRating]);
‚úÖ Works
Now isTop becomes correct after the movie data arrives because the effect re-runs when the rating updates.

3) Best Option (Most cases): Use derived state instead of real state

In this case, isTop is fully determined by other data (imdbRating). That means it‚Äôs typically better as derived state. 

// ‚úÖ Derived state: recomputed every render
const isTop = Number(imdbRating) > 8;
‚úÖ Why derived state is powerful
  • No extra state to manage.
  • No extra effect needed.
  • Always up-to-date on every render.
üí° Rule of thumb
If a value can be calculated from existing props/state, prefer derived state instead of storing it in useState.

4) State Updates ‚ÄúFeel Async‚Äù: You can‚Äôt read updated state immediately

Next, the lecture demonstrates a very important behavior: after calling a state setter (like setAvgRating(...)), you cannot assume the state variable changed immediately within the same event handler.

Experiment: set state then read it right away

const [avgRating, setAvgRating] = useState(0);

function handleAdd() {
  setAvgRating(Number(imdbRating));
  alert(avgRating); // üö® still shows the old value
}
‚ö†Ô∏è What‚Äôs happening
  • React batches updates and applies them after your event handler finishes.
  • So inside the same handler, avgRating is still the previous value.
  • This is why it looks ‚Äúasynchronous.‚Äù
Term Meaning
Stale state When you read a state variable that hasn‚Äôt been updated yet (because React hasn‚Äôt applied the update).
Batching React groups multiple state updates together for performance and applies them in one go.

5) When you must use the callback form of setState

If your new state depends on the previous state, always use the callback form: setState(prev => next).

‚ùå Buggy version (uses stale avgRating)

// ‚ùå avgRating might still be 0 here (stale)
setAvgRating((avgRating + userRating) / 2);

Example: if avgRating is still 0 and userRating is 10, you get 5 (wrong).

‚úÖ Correct version (callback gets the current value)

// ‚úÖ React provides the most up-to-date value
setAvgRating(currentAvg => (currentAvg + userRating) / 2);
‚úÖ Why this works
React passes the latest state value into the callback, so you never compute from stale data.
üß† Memory tip
If you see yourself writing setX(x + 1), pause and ask: ‚ÄúCould x be stale here?‚Äù If yes, use setX(prev => prev + 1).

6) Final recap (what to remember)

1 The value passed to useState(initial) matters only on mount.

2 If a value can be calculated from other state/props, prefer derived state.

3 State updates aren‚Äôt immediately reflected within the same handler (batching).

4 If the next state depends on the previous state, use the callback form: setX(prev => ...).
