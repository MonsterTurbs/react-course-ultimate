2) Checkbox as a Controlled Element ‚òëÔ∏è

A controlled input means React controls its value. For a checkbox, you control the checked prop. (Not value ‚Äî checkboxes use checked.)

Item: add the checkbox

function Item({ item, onToggleItem }) {
  return (
    <li>
      <input
        type="checkbox"
        checked={item.packed}
        onChange={() => onToggleItem(item.id)}
      />

      <span style={item.packed ? { textDecoration: "line-through" } : {}}>
        {item.quantity} {item.description}
      </span>

      <button>‚ùå</button>
    </li>
  );
}

The checkbox is controlled because its checked state is always derived from item.packed, and the onChange triggers an update.

If you set checked={item.packed} but forget onChange, the checkbox becomes ‚Äúread-only‚Äù in practice. React is forcing the checkbox to match state, but nothing updates the state.

3) Create the Toggle Handler Where State Lives (in App) üè†

We want to update exactly one object inside an array. The standard immutable pattern is: map the array and replace the matching object with a new object. 

function App() {
  const [items, setItems] = useState([]);

  function handleToggleItem(id) {
    setItems((items) =>
      items.map((item) =>
        item.id === id
          ? { ...item, packed: !item.packed }
          : item
      )
    );
  }

  return (
    <div className="app">
      <PackingList
        items={items}
        onDeleteItem={handleDeleteItem}
        onToggleItem={handleToggleItem}
      />
    </div>
  );
}
üß±

Why map + spread?

map() returns a new array. { ...item } returns a new object. This avoids mutating state and keeps React updates predictable.

Toggle logic: packed: !item.packed flips the boolean. true ‚Üí false, false ‚Üí true.

4) Pass the Toggle Handler Down to Item ‚¨áÔ∏è

Since Item is rendered inside PackingList, we forward the handler through it.

PackingList forwards onToggleItem

function PackingList({ items, onDeleteItem, onToggleItem }) {
  return (
    <div className="list">
      <ul>
        {items.map((item) => (
          <Item
            key={item.id}
            item={item}
            onDeleteItem={onDeleteItem}
            onToggleItem={onToggleItem}
          />
        ))}
      </ul>
    </div>
  );
}

This is the common React pattern: parent owns the state, child triggers updates by calling a callback prop.

5) Common Pitfalls (and Fixes) üßØ

Pitfall A: Using value instead of checked

// ‚ùå Not the typical controlled checkbox pattern
<input type="checkbox" value={item.packed} />

// ‚úÖ Correct
<input type="checkbox" checked={item.packed} />

Pitfall B: Calling the handler immediately

// ‚ùå Runs immediately on render
onChange={onToggleItem(item.id)}

// ‚úÖ Pass a function, React calls it later
onChange={() => onToggleItem(item.id)}

Pitfall C: Mutating the object directly

// ‚ùå Don't do this (mutation)
setItems((items) => {
  const item = items.find((i) => i.id === id);
  item.packed = !item.packed;
  return items;
});

// ‚úÖ Do this (immutable)
setItems((items) => items.map((item) =>
  item.id === id ? { ...item, packed: !item.packed } : item
));

6) What React Does After Toggling üîÑ

  1. User clicks checkbox in Item.
  2. Item calls onToggleItem(item.id).
  3. App updates items using setItems.
  4. React re-renders ‚Üí checkbox state updates ‚Üí strike-through updates.

Notice the mental model: we don‚Äôt ‚Äúset a class in the DOM.‚Äù We change state and React keeps the UI in sync.

7) Quick Self-Check üß†

  1. Why does handleToggleItem live in App and not in Item?
  2. Why do we use map() for update but filter() for delete?
  3. Why do we need { ...item } when toggling?
  4. Why does a checkbox use checked (not value)?

Answer hints ‚úÖ

  • The state array lives in App; Item only renders one element and triggers events.
  • filter() removes elements; map() keeps length but changes one element.
  • To avoid mutating the existing object in state (immutability).
  • Checkbox UI is controlled by its checked state (true/false).
