üß† Core concept: styled-components uses template literals

Styled components are written with tagged template literals. That means you can interpolate JavaScript inside the CSS using ${...}.

Key idea: Anything inside ${...} is JavaScript. It can return strings (CSS) conditionally.

Example: conditional CSS value

import styled from "styled-components";

const Heading = styled.h1`
  font-size: ${10 > 5 ? "30px" : "5px"};
`;

This is just a demo to prove you can use JS. In real code, you‚Äôd use props (next sections).

üß± Extract CSS blocks into variables (and why css helps)

Sometimes you want to store a chunk of CSS in a variable and inject it into multiple components. A plain string works, but the editor may lose CSS syntax highlighting.

Basic variable (can lose highlighting)

const centered = `
  text-align: center;
`;

const Heading = styled.h1`
  ${centered}
`;

Using the styled-components css helper ‚úÖ

import styled, { css } from "styled-components";

const centered = css`
  text-align: center;
`;

const Heading = styled.h1`
  ${centered}
`;
Why css? It preserves editor highlighting and is designed for reusable CSS with interpolations.
The transcript notes: some cases may still ‚Äúwork‚Äù without css, but the docs recommend css when you generate CSS from a template literal with interpolations.

üéõÔ∏è Props-driven styles (the real goal)

The practical reason we want JS inside the template literal is to read props and decide which CSS to apply. In styled-components, you can interpolate a function: 

${props => /* return CSS */}

üìÑ File: ui/Heading.jsx (initial version)

import styled, { css } from "styled-components";

const Heading = styled.h1`
  line-height: 1.4;

  ${props =>
    props.as === "h1" &&
    css`
      font-size: 3rem;
      font-weight: 600;
    `}

  ${props =>
    props.as === "h2" &&
    css`
      font-size: 2rem;
      font-weight: 600;
    `}

  ${props =>
    props.as === "h3" &&
    css`
      font-size: 2rem;
      font-weight: 500;
    `}
`;

export default Heading;
Pattern: each block is ‚Äúconditionally injected‚Äù only when the condition is true.

‚ö†Ô∏è Big gotcha: styling is not the same as semantic HTML

If you only pass your own prop like type="h2" and use it to change styles, the DOM element might still be <h1>. That looks right visually, but the markup is wrong.

Why it matters: wrong heading tags can harm accessibility (screen readers) and SEO (document structure).

Bad approach (visual only)

// ‚ùå Still renders an <h1> in the DOM
<Heading type="h2">Check-in and out</Heading>

Correct approach: use the as prop ‚úÖ

Styled-components supports a special prop called as that changes the rendered HTML tag. So the DOM becomes semantically correct. 

// ‚úÖ Renders an actual <h2>
<Heading as="h2">Check-in and out</Heading>
Takeaway: Use as for semantics; use your conditional CSS blocks to match styling for that level.
