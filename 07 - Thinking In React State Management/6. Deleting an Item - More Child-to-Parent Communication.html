1) Goal: Click ‚ùå to Remove an Item

  • Each packing entry is rendered by an Item component.
  • The delete click happens in Item (child).
  • The items state lives in App (parent).

This is another classic case of child-to-parent communication: a child component needs to trigger a state update that lives in the parent.

2) Step 1 ‚Äî Create the Delete Handler Where State Lives (in App) üè†

We delete by updating state. To know which element to delete, we pass an id into the handler. 

function App() {
  const [items, setItems] = useState([]);

  function handleDeleteItem(id) {
    setItems((items) => items.filter((item) => item.id !== id));
  }

  return (
    <div className="app">
      <Form onAddItems={handleAddItems} />
      <PackingList items={items} onDeleteItem={handleDeleteItem} />
      <Stats items={items} />
    </div>
  );
}

Why filter? It returns a new array (immutable). Anything that doesn‚Äôt match stays. The matching id is removed.

üß±

Immutable delete pattern

Delete with filter() so you don‚Äôt mutate the original array (React-friendly).

3) Step 2 ‚Äî Pass the Handler Down to Where the Click Happens ‚¨áÔ∏è

The click is inside Item, but Item is created inside PackingList. So the handler travels through PackingList to each Item.

PackingList receives and forwards the handler

function PackingList({ items, onDeleteItem }) {
  return (
    <div className="list">
      <ul>
        {items.map((item) => (
          <Item
            key={item.id}
            item={item}
            onDeleteItem={onDeleteItem}
          />
        ))}
      </ul>
    </div>
  );
}

PackingList doesn‚Äôt "use" the delete handler itself; it just passes it along.

You can‚Äôt pass from App directly to Item unless App renders Item directly. Props flow only through the actual parent‚Üíchild relationships.

4) Step 3 ‚Äî Call the Handler Correctly in Item (Don‚Äôt Accidentally Pass the Event) üéØ

In Item, we want to call onDeleteItem with item.id. If you pass onDeleteItem directly to onClick, React calls it with the event object.

Common mistake

This passes the click event object into your function (not the id).

Wrong ‚ùå

function Item({ item, onDeleteItem }) {
  return (
    <li>
      <span>{item.quantity} {item.description}</span>
      <button onClick={onDeleteItem}>‚ùå</button>
    </li>
  );
}
‚úÖ

Correct pattern

Wrap the call in an arrow function so you can pass your own argument (item.id).

Correct ‚úÖ

function Item({ item, onDeleteItem }) {
  return (
    <li>
      <span>{item.quantity} {item.description}</span>
      <button onClick={() => onDeleteItem(item.id)}>‚ùå</button>
    </li>
  );
}

Why this works: you‚Äôre passing a function to React. React calls it on click, and then your function calls onDeleteItem with the id.

This is the same principle as form submission: React calls your handler with an event ‚Äî unless you wrap and pass what you need (like an id).

5) What React Does After Delete üîÑ

  1. User clicks ‚ùå in Item.
  2. Item calls onDeleteItem(item.id) ‚¨ÜÔ∏è
  3. App runs setItems with a filtered array.
  4. State updates ‚Üí React re-renders affected components ‚Üí UI updates (item disappears). ‚ú®

Key takeaway: the UI changed because state changed. We did not manually remove DOM nodes. React re-rendered the view.

6) Bonus Note: Initial Items vs Empty State üß™

You can initialize your state with sample data (helpful for demos/testing) or keep it empty: 

// Empty by default

// Or preload test items

If ESLint warns that initialItems is unused, remove it or use it as the initial state.

Answer hints ‚úÖ
