üéØ Goal of this step

You will take your local Next.js project, ensure it is properly tracked by Git, then push it to a new GitHub repository. After that, the repo can be connected to Vercel for continuous deployment.

‚úÖ Success looks like this
  • Local working directory is clean (no unwanted uncommitted files)
  • At least one commit exists (e.g., Initial commit)
  • GitHub shows your project files after refresh
  • You have a remote named origin pointing to GitHub

üß≠ Big picture: why GitHub matters for Vercel

üèóÔ∏è Vercel deployment flow
  1. Push code to GitHub
  2. Connect the GitHub repo to Vercel
  3. Vercel builds + deploys on every push (CI/CD)
ü§ù Why this is ‚Äúprofessional‚Äù
  • Repeatable: anyone can clone your repo and build the same project
  • Traceable: commits show history (who changed what, when)
  • Automated: Vercel can redeploy automatically on new commits

üåê Create a new GitHub repository

Create a new repository on GitHub (usually Private while learning, unless you want it public).

  • Give it a valid repo name (e.g., the-wild-oasis-website)
  • Do not initialize with README / .gitignore / license if your local repo already has them (avoids conflicts)
Choice Recommendation Why
Visibility Private (initially) Keeps credentials and work-in-progress code safer while learning
Initialize with README No You already have a local history‚Äîavoid merge conflicts on first push
.gitignore No Create-Next-App already created one

‚¨ÜÔ∏è Add remote + push to GitHub

Now connect your local repo to GitHub using a remote named origin, then push. GitHub will show you commands after creating the repo.

Typical commands (existing repo ‚Üí push)

git push -u origin main
üñºÔ∏è Why pushing might take time

Large assets (images, videos, etc.) increase push time. This is normal.

Verify the remote

# origin  <YOUR_GITHUB_REPO_URL> (fetch)
# origin  <YOUR_GITHUB_REPO_URL> (push)

After pushing, refresh the GitHub repo page and confirm your code is visible. Once that‚Äôs working, you‚Äôre ready to connect the repo to Vercel.

üßØ Common pitfalls + quick fixes

‚ùå ‚Äúremote origin already exists‚Äù

You already added a remote earlier.

git remote -v
# If the URL is wrong:
git remote set-url origin <YOUR_GITHUB_REPO_URL>
‚ùå First push rejected / conflicts

Usually caused by initializing the GitHub repo with README/.gitignore.

# Option A (recommended): delete the GitHub repo and recreate it WITHOUT initializing files

# Option B: pull then push (requires resolving conflicts)
git pull origin main --allow-unrelated-histories
‚ùå Authentication errors (HTTPS)

GitHub may require a Personal Access Token (PAT) instead of your password.

  • Use a PAT when prompted for password
  • Or switch to SSH if you prefer
‚ùå Accidentally committed secrets

Never commit real credentials. Use environment variables.

  • Confirm .env.local is in .gitignore
  • If leaked, rotate keys and remove from Git history (advanced)
