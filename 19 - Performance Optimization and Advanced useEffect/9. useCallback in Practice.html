üéØ Goal of this lecture

Use useCallback to memoize a function so that a memoized component (via memo) doesn‚Äôt re-render unnecessarily when the parent re-renders.

‚úÖ Big idea:

If a memoized child receives a new function prop every render, React treats it as ‚Äúprops changed‚Äù and memo will not bail out. useCallback makes that function stable across renders.

üß© Quick refresher: Why memo breaks with functions

  • In React, when a component re-renders, everything inside it is recreated (including functions).
  • In JavaScript, functions are objects. A ‚Äúnew‚Äù function is a different identity, even if code looks the same.
  • So a memoized child sees: new function prop ‚áí props changed ‚áí child re-renders.
‚ö†Ô∏è Symptom you can feel:

The UI becomes laggy (typing delays, clicking delays) because a heavy component (like Archive) keeps re-rendering.

üõ†Ô∏è The setup: bring back onAddPost into a memoized Archive

The lecture reintroduces a button inside Archive by passing a callback prop: onAddPost={handleAddPost}.

  archiveOptions={archiveOptions}
  onAddPost={handleAddPost}

// Child (Archive) ‚Äî memoized
const Archive = memo(function Archive({ archiveOptions, onAddPost }) {
  return (
    <div>
      <h2>{archiveOptions.title}</h2>
      <button onClick={onAddPost}>Add post</button>
      {/* ...big list... */}
    </div>
  );
});

Now toggle some parent state (example: a fake dark mode button). Even though Archive is memoized, it starts re-rendering again because handleAddPost is recreated on every parent render.

‚úÖ The fix: memoize the function with useCallback

Wrap the function definition in useCallback. This makes the function identity stable as long as dependencies don‚Äôt change.

Before (unstable function prop) üê¢

  // updates state
  setPosts((posts) => [newPost, ...posts]);
}

<Archive onAddPost={handleAddPost} />

Problem: each render creates a new handleAddPost.

After (stable function prop) üöÄ

  setPosts((posts) => [newPost, ...posts]);
}, []);

<Archive onAddPost={handleAddPost} />

Result: Archive can skip re-renders if other props are stable.

üß† How to remember the difference:
  • useMemo caches a value (result of a function).
  • useCallback caches the function itself (its identity).

üîó Dependency arrays and ‚Äústale closures‚Äù

useCallback has a dependency array just like useEffect and useMemo. If you reference values inside the callback (like posts), you may need them in the dependencies.

Preferred pattern: use the setter callback ‚úÖ

// Avoid needing `posts` in dependencies by using the functional update

  setPosts((posts) => [newPost, ...posts]);
}, []);
‚ö†Ô∏è If you reference outer variables:

Example: using posts directly inside the callback can create a stale closure if not added as a dependency. ESLint will usually warn you.

üßØ Don‚Äôt overuse useCallback / useMemo

  • These hooks have a cost: React must store cached values/functions and compare dependencies.
  • They make sense when they prevent a real performance problem (visible lag) or when a component is heavy and re-renders frequently.
  • Don‚Äôt wrap ‚Äúeverything‚Äù by default‚Äîoptimize with evidence (Profiler), not guesswork.
üîé Practical rule of thumb:

Use memo for a heavy component; then use useMemo/useCallback only when you must stabilize objects/functions passed into that memoized component.
