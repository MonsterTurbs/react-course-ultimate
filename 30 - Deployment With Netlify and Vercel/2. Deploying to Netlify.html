üß† Big Picture (What you are doing and why)

Deploying a React SPA usually means: you ship the static build output (HTML/CSS/JS/assets) to a web host. The browser downloads that bundle and runs the app.

In this lecture, you do a manual deploy (drag-and-drop) to Netlify ‚Äî fast and simple for demos/portfolios.

Security reminder üßπ Before deploying, remove any hard-coded credentials or default values left in your login form. Otherwise, anyone who finds your site can log in.

Key output: the dist folder created by npm run build. That folder is what you upload to Netlify.

‚úÖ Step-by-step Checklist (Do this in order)

  1. Clean login defaults (remove saved credentials / prefilled values).
  2. Create production build: run npm run build (Vite bundles your app into the production output).
  3. Fix lint/build warnings if they block the build or are easy to clean:
    • Unused variables (example: set disabled state when creating).
    • Unused imports (example: remove unused date utilities / fake data if no longer needed).
  4. Confirm build output: check dist/ contains:
    • index.html
    • assets/ (your JS bundle, CSS, etc.)
    • images/other static files if used
  5. Add SPA redirect config (Netlify needs it for React Router routes).
  6. Netlify manual deploy: drag-and-drop the dist folder into Netlify‚Äôs manual deploy UI.
  7. Open deployed URL, log in with your own credentials, and verify key flows.
  8. Optional: change Netlify site name to get a nicer URL.

üì¶ Building the Production Bundle (Vite)

Vite takes your development files and creates an optimized production build. You run:

npm run build

Beginner-friendly explanation üß©

  • During development, Vite serves files quickly and supports hot reload.
  • For production, it bundles and optimizes your code into files the browser can load efficiently.
  • The result is a folder (usually dist) containing static files you can host anywhere.

Large bundle warning ‚ö†Ô∏è Vite may warn if your JS bundle is large. In this project, it‚Äôs acceptable because the app is behind login and has a small user base. In other apps, you might use code splitting to reduce initial load.

üßπ Cleaning Common Build Issues (Examples)

Typical issues you may see during build:

  • ESLint warnings about unused variables/imports.
  • Leftover fake data used during chart development.
Example fixes from the lecture (click to expand)
  • Cabin row: unused variable ‚Üí use it (e.g., disable button while creating) or remove it.
  • Sales chart: remove unused imports and delete fake chart data once real data is used.

Tip: keep builds ‚Äúquiet‚Äù to reduce risk and improve maintainability.

üåê Netlify Manual Deploy (Drag-and-Drop)

Netlify can host static sites (including React SPA builds) very easily.

  1. Create a Netlify account and log in.
  2. Choose Deploy manually (manual site deploy).
  3. Drag-and-drop your dist folder.
  4. Open the production deploy URL and verify the app.

Reality check ‚úÖ If your app works locally in production preview/build and you upload the correct dist folder, Netlify should serve it just like your local build.

üß≠ Important: Single Page App Redirects (React Router on Netlify)

If your app uses client-side routing (React Router), Netlify needs to redirect all routes to index.html, so React can handle the route in the browser.

Symptom üòµ If you refresh a deep route like /account or open it directly, you might see a 404 on Netlify without proper redirects.

Create a file named netlify.toml with this redirect rule:

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200

Where should this file live? üìÅ

  • Best option: put it in public/ so it gets copied into dist/ on every build.
  • If you place it directly in dist/, it may be deleted next time you rebuild.

üîó Shareable URL + Site Name

Netlify gives you a generated URL, but you can rename your site to get a nicer subdomain.

  • Go to Site settings ‚Üí change the site name.
  • Optional: attach a custom domain you own.

Sharing access üîê Remember: since the app is behind login, your friends/employers can only explore it if you provide credentials (or if you build a public demo mode later).

‚ôªÔ∏è Updating the Deployment (Manual)

If you change the code later:

  1. Update the code locally.
  2. Run npm run build again.
  3. Ensure netlify.toml is still included (ideally via public/).
  4. Drag-and-drop the new dist folder into Netlify again (Deploys page).

üèóÔ∏è Next Step Preview: Continuous Deployment (Why GitHub matters)

Manual deploy is perfect for quick demos. A more ‚Äúreal-world‚Äù workflow is continuous deployment:

  • Push code to a GitHub repository.
  • Connect the repo to a host (Netlify or Vercel).
  • Every push automatically triggers a new build + deploy.

Benefit ‚úÖ No more drag-and-drop. Your deploy always matches your repository history.

üßæ Quick Glossary

  • Build: Creating optimized static files for production (Vite output).
  • Bundle: Combined JS/CSS output produced by the build tool.
  • SPA: Single Page Application ‚Äî routes are handled in the browser.
  • Redirect to index.html: Host-level rule so React Router can resolve routes.
  • Continuous Deployment: Auto-deploy on every push to Git.
