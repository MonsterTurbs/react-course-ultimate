üß© Why we need getStaticProps in the Pages Router

In the App Router, you could fetch data directly inside a Server Component (e.g., make the component async and fetch at the top level).

In the Pages Router, pages are regular React components ‚Äî and React components:

  • ‚ùå Can‚Äôt do top-level server data fetching the same way as Server Components.
  • ‚ùå If you fetch in the component normally, it tends to happen on the client (after hydration), which defeats server pre-rendering.
üí° So Next.js introduced special server-side data APIs for the Pages Router: getStaticProps and getServerSideProps.

üõ†Ô∏è The recipe: Fetch cabins with getStaticProps

Step-by-step flow (what Next.js does):

  1. Next.js runs getStaticProps on the server.
  2. You fetch your data there (e.g., getCabins() from your data service).
  3. You return the data inside an object: { props: { ... } }.
  4. Next.js passes those props into your page component.
  5. The page is generated as HTML (static generation) and sent to the browser.

Example skeleton (minimal):

// pages/cabins.js
import CabinList from "../components/CabinList";
import { getCabins } from "../lib/data-service";

export default function Cabins({ cabins }) {
  return (
    <main>
      <h1>Cabins</h1>
      <CabinList cabins={cabins} />
    </main>
  );
}

export async function getStaticProps() {
  // ‚úÖ runs on the server
  const cabins = await getCabins();

  return {
    props: {
      cabins, // must match the prop name used above
    },
  };
}

üìå Why the name ‚ÄúStatic Props‚Äù?

It‚Äôs called getStaticProps because it‚Äôs used for Static Site Generation (SSG). That means the page is generated statically (not per-request).

üèóÔ∏è Pages Router rule:

This is more ‚Äúexplicit‚Äù than the App Router. In the Pages Router, you are clearly choosing the strategy by which function you use.

üñºÔ∏è Common gotcha: Next.js Image hostname config

‚ö†Ô∏è Important: the hostname should NOT include https://. Use only the host.
