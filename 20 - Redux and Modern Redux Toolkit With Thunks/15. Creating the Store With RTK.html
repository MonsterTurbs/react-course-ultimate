üß† Big idea: upgrade gradually (no big rewrite)

RTK is 100% compatible with classic Redux. That means you can:

  • ‚úÖ Keep your existing reducers/slices exactly as they are,
  • ‚úÖ Then refactor slices later (optional, when you‚Äôre ready).
‚úÖ This is a safe migration strategy: fewer moving parts, easier debugging.

üèóÔ∏è Step 2: Replace createStore with configureStore

In classic Redux, store setup often includes: combineReducers, applyMiddleware, thunk, and DevTools wiring. RTK compresses that into a single call.

Old (classic Redux) mental model

‚ö†Ô∏è Lots of setup code (boilerplate). Easy to forget a step (thunk, DevTools, etc.). 
// Classic Redux (conceptual example)
import { createStore, combineReducers, applyMiddleware } from "redux";
import thunk from "redux-thunk";
import { composeWithDevTools } from "redux-devtools-extension";

const rootReducer = combineReducers({
  account: accountReducer,
  customer: customerReducer,
});

const store = createStore(
  rootReducer,
  composeWithDevTools(applyMiddleware(thunk))
);

export default store;

New (Redux Toolkit) approach

‚úÖ configureStore automatically:
  • combines reducers
  • adds thunk middleware
  • enables Redux DevTools (in development)
// Redux Toolkit (RTK)
import { configureStore } from "@reduxjs/toolkit";

import accountReducer from "../features/accounts/accountSlice";
import customerReducer from "../features/customers/customerSlice";

const store = configureStore({
  reducer: {
    account: accountReducer,
    customer: customerReducer,
  },
});

export default store;

‚úÖ Note: Your React side stays the same ‚Äî <Provider store={store}> is unchanged.

üîç What exactly does configureStore do for you?

1) Combines reducers automatically üß©

You pass an object of reducers ‚Üí RTK makes a root reducer under the hood.

2) Adds Thunk middleware by default ‚è≥

So async action creators (thunks) work without manual applyMiddleware(thunk).

3) Enables Redux DevTools üïµÔ∏è

No need for composeWithDevTools setup in a typical RTK app.

‚ö†Ô∏è RTK wraps classic Redux. It‚Äôs not ‚Äúmagic‚Äù‚Äîit‚Äôs best-practice defaults packaged neatly.

üß© ‚ÄúReducers choose the state shape‚Äù (important!)

The keys you use in the reducer object become the top-level keys in your store state.

  reducer: {
    account: accountReducer,
    customer: customerReducer,
  }

// State shape becomes:
// {
//   account: { ... },
//   customer: { ... }
// }
‚úÖ That‚Äôs why selectors look like: state.account.balance or state.customer.fullName.

üö´ Common mistakes and fixes

‚ùå Mistake: Wrong reducer import path (feature folder moved)

Fix: Ensure the slice file exports the reducer correctly (often default export), and update the import path.

‚ùå Mistake: Expecting createStore options to work the same

Fix: RTK uses an options object; pass reducers under reducer.

‚ö†Ô∏è Note: You still need react-redux for React integration.

RTK handles Redux setup; React components still talk to the store using Provider, useSelector, useDispatch.
