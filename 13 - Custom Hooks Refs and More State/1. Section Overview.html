3) Quick refresher: What are Hooks? üß©

Hooks are functions that let function components ‚Äúhook into‚Äù React features‚Äîlike state, refs, and effects. Instead of writing classes, you compose behavior using functions.

Common Hooks you already know / will use here

  • üß† useState ‚Äî store and update component state
  • üß∑ useRef ‚Äî store a mutable value that persists across renders (without re-rendering)
  • üß∞ Custom Hook ‚Äî extract reusable logic into a function that starts with use

4) Rules of Hooks (the ‚Äúwhy‚Äù + the ‚Äúhow‚Äù) üìè

üö´ The two big rules

  • Only call Hooks at the top level (not inside loops, conditions, or nested functions).
  • Only call Hooks from React functions (function components or custom Hooks).

Why does this exist? React relies on the order of Hook calls to correctly match state/ref ‚Äúslots‚Äù between renders. If you call Hooks conditionally, the order changes and React can‚Äôt reliably associate data with the right Hook call.

‚úÖ Correct pattern

function MyComponent({ show }) {
  const [count, setCount] = React.useState(0);

  // ‚úÖ Condition comes AFTER hooks
  if (!show) return null;

  return (
    <button onClick={() => setCount(c => c + 1)}>
      Clicked {count} times
    </button>
  );
}

‚ùå Incorrect pattern

function MyComponent({ show }) {
  // ‚ùå Hook inside a condition
  if (show) {
    const [count, setCount] = React.useState(0);
  }
  return null;
}

Result: Hook call order becomes unpredictable ‚Üí bugs or runtime errors.

5) useState deeper understanding üß†

useState is more than ‚Äústore a value.‚Äù The goal is to model UI state cleanly and update it safely.

Helpful patterns

  • Functional updates (best when the next state depends on the current state):
setCount(c => c + 1);
  • Derived state: don‚Äôt store what you can compute (avoid duplicate sources of truth).
  • Single responsibility state: keep state minimal and focused (better debugging + fewer bugs).

Rule of thumb: If you can calculate it from props/state during render, you probably don‚Äôt need to store it in state.

6) useRef basics üß∑

useRef gives you a container that persists between renders without triggering a re-render when it changes.

Two common use cases

  • üéØ DOM access (focus an input, measure an element, etc.)
  • üóÉÔ∏è Instance-like storage (store a timeout id, previous value, mutable flags, etc.)
function SearchBox() {
  const inputRef = React.useRef(null);

  function focusInput() {
    inputRef.current?.focus();
  }

  return (
    <div>
      <input ref={inputRef} placeholder="Type..." />
      <button onClick={focusInput}>Focus</button>
    </div>
  );
}

‚ö†Ô∏è What useRef is NOT

  • Not a replacement for state when UI must update.
  • If the screen needs to reflect the change, you usually want useState.

7) Custom Hooks üß∞

A custom Hook is a function that starts with use and can call other Hooks. It extracts reusable behavior so multiple components can share logic without duplication.

‚úÖ Example: simple custom Hook

function useToggle(initial = false) {
  const [value, setValue] = React.useState(initial);
  const toggle = React.useCallback(() => setValue(v => !v), []);
  return [value, toggle];
}

function Demo() {
  const [isOpen, toggle] = useToggle();
  return (
    <button onClick={toggle}>
      {isOpen ? "Open" : "Closed"}
    </button>
  );
}

Custom Hooks should feel like a ‚Äúmini API‚Äù for one purpose. If you find the hook doing unrelated tasks, split it.
