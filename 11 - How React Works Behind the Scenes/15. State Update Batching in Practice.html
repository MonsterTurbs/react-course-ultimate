3) State updates are asynchronous (stale state demo)

The lecture demonstrates this by logging likes right after calling setLikes(0).

‚ö†Ô∏è Key observation

üß† Beginner-friendly explanation

4) ‚ÄúWhy no re-render?‚Äù when setting the same state

The lecture asks: if you click Undo again (when it‚Äôs already reset), why no ‚Äúrender‚Äù log?

‚úÖ Rule

If the new state equals the current state, React can skip re-rendering that component instance. Example: showDetails already true and likes already 0.

üîç Practical takeaway

If you expect UI changes but nothing happens, verify that your ‚Äúnew‚Äù state value is actually different.

5) Triple increment bug: why setLikes(likes + 1) fails 3√ó

5.1 The ‚Äúwrong‚Äù version (for learning)

function handleTripleInc() {
  setLikes(likes + 1);
  setLikes(likes + 1);
  setLikes(likes + 1);
}
‚ùå What you observe

Likes increases by only 1 instead of 3.

üß† Why it happens

All three lines read the same stale likes snapshot from the current render. React queues three updates that all compute ‚ÄúoldLikes + 1‚Äù, so the final result is still just +1.

5.2 The correct version: functional updates

function handleTripleInc() {
  setLikes((cur) => cur + 1);
  setLikes((cur) => cur + 1);
  setLikes((cur) => cur + 1);
}
‚úÖ Why this works

The callback receives the latest value in the update queue: 0 ‚Üí 1 ‚Üí 2 ‚Üí 3.

‚ö†Ô∏è Rule you should memorize

Whenever new state depends on previous state, always use: setState(cur => next).

5.3 ‚ÄúBut it works now‚Ä¶ why be strict?‚Äù

Even if setLikes(likes + 1) works today, it can break later when someone calls your function multiple times or adds more updates. Functional updates keep you safe as code evolves.

6.2 Historical note: React 17 behavior

üï∞Ô∏è Before React 18

Automatic batching used to apply mainly to React event handlers. Updates in timeouts/promises could cause multiple renders.

üôÇ What you should remember

In modern React (18+), batching is more consistent ‚Äúeverywhere,‚Äù which improves performance and reduces surprise re-renders.

8) Mini-Quiz (with answers)

Q1) In handleUndo, why does console.log(likes) print the old value?

Answer: Because state updates are applied on the next render. Until React re-renders, the current render‚Äôs state snapshot is still in scope (stale state).

Q2) Why does setLikes(likes + 1) called 3 times only add +1?

Answer: All three calls read the same stale likes value, so they queue the same update (oldLikes + 1). Use functional updates instead.

Q3) When can React skip a re-render?

Answer: When you set state to a value that is equal to the current state (no actual change), React may bail out and skip rendering.

Q4) What is the safest pattern for updating state based on previous state?

Answer: setState(cur => computeNext(cur)) (functional updates).
