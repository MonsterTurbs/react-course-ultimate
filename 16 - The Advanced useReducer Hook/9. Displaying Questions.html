üéØ Goal of this lecture

Render only 1 question at a time (not all questions).

We keep track of the ‚Äúcurrent question‚Äù using an index stored in state.

state.index questions[index] <Question /> <Options />

1) Why do we need an index state?

‚úÖ Because when the user goes to the next question, we must re-render the UI. Any value that changes the UI must live in state.

üí° Mental model
  • questions is the full array (e.g., 15 questions).
  • index is a pointer to the current question.
  • Question #1 is at questions[0] ‚Üí so index starts at 0.
  • Next question = increment index from 0 ‚Üí 1 ‚Üí 2‚Ä¶

‚úÖ Add it to initialState

// reducer state (example)
const initialState = {
  questions: [],
  status: "loading",
  index: 0, // üëà NEW: current question index
};

‚ö†Ô∏è Common pitfall: If you forget to destructure index from state, you‚Äôll get ‚Äúindex is not defined‚Äù or you‚Äôll render the wrong question.

2) Send the current question to the <Question /> component

We compute the ‚Äúcurrent question‚Äù using questions[index], then pass it as a prop.

‚úÖ In App.js (active screen)

// Example shape in App component
const [{ questions, status, index }, dispatch] = useReducer(reducer, initialState);

// ...inside JSX
{status === "active" && (
  <Question question={questions[index]} />
)}
üîç Beginner note

questions[index] means: ‚Äúgive me the question object at position index.‚Äù

At the start of the quiz, index = 0 ‚Üí you show the first question.

‚úÖ Safety guard (optional but professional)

If there is a chance questions is still empty (e.g., slow network), add a guard:

{status === "active" && questions.length > 0 && (
  <Question question={questions[index]} />
)}

3) Render the question text

Inside Question, you receive a question prop. That prop is an object with a shape like:

// question object example
{
  question: "React is a ...?",
  options: ["Library", "Framework", "Database", "Compiler"],
  correctOption: 0,
  points: 10
}

‚úÖ In Question.js

export default function Question({ question }) {
  return (
    <div>
      <h4>{question.question}</h4>
      {/* options will be rendered next */}
    </div>
  );
}
üìù Why question.question looks weird?

The prop name is question, and inside it there‚Äôs a property also named question (the text). You could rename it while destructuring if you want:

export default function Question({ question: currentQuestion }) {
  return <h4>{currentQuestion.question}</h4>;
}

4) Render options as buttons

Instead of using a list (<ol>/<li>), we render buttons because each option is clickable.

‚úÖ Map over question.options

<div className="options">
  {question.options.map((option) => (
    <button
      key={option}
      className="btn btn-option"
      onClick={() => {/* handle answer later */}}
    >
      {option}
    </button>
  ))}
</div>
‚úÖ Key rule

Each mapped element needs a stable key. Here we use the option string because options are unique.

‚ö†Ô∏è Common pitfall

If two options could be identical, use the index instead:

{question.options.map((option, i) => (
  <button key={i} ...>{option}</button>
))}

5) Split Question into smaller components

üß© A clean practice: when a component has a clear sub-part (like ‚Äúoptions list‚Äù), extract it into its own component.

‚úÖ Create Options.js

export default function Options({ question }) {
  return (
    <div className="options">
      {question.options.map((option) => (
        <button key={option} className="btn btn-option">
          {option}
        </button>
      ))}
    </div>
  );
}

‚úÖ Use it inside Question.js

import Options from "./Options";

export default function Question({ question }) {
  return (
    <div>
      <h4>{question.question}</h4>
      <Options question={question} />
    </div>
  );
}
üéØ Benefit

Question becomes easier to read: it shows the structure (‚Äúquestion text + options‚Äù), while the details live in Options.

6) Move components into a /components folder

üìÅ As the app grows, it‚Äôs common to create a components folder and move component files there.

‚úÖ What changes when you move files?

  • App import in index.js must be updated because App‚Äôs path changed.
  • Imports inside App.js usually still work if they‚Äôre in the same folder as the components they import (or you update them accordingly).

Example: Update index.js

// before
import App from "./App";

// after (if App is moved into /components)
import App from "./components/App";
üß† Quick check

If the screen goes blank after moving files, it‚Äôs usually an import path issue.
