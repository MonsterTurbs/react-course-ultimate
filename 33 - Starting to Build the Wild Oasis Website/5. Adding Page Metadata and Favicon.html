1) üß† Big idea: Next.js metadata is convention-driven

In the App Router, Next.js has a built-in API for metadata. Instead of manually writing <title> and <meta> tags inside a custom HTML head, you typically export a metadata object from a layout.js or page.js.

‚úÖ Why this matters

It centralizes SEO-related data, works naturally with routing/layouts, and keeps your app aligned with Next.js conventions.

2) üß© Root layout vs per-page metadata

üåç Root layout metadata (global default)

Export metadata in app/layout.js to apply it across pages by default.

// app/layout.js
export const metadata = {
  title: "The Wild Oasis",
};

export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}

üìÑ Page metadata (override)

Export metadata in app/cabins/page.js to override the layout default.

// app/cabins/page.js
export const metadata = {
  title: "Cabins",
};

export default function Page() {
  return <h1>Cabins</h1>;
}

üìù Key rule

Page-level metadata overrides layout metadata for that route.

3) üè∑Ô∏è Better titles: template + default (real-world pattern)

A common website pattern is: PageName ¬∑ SiteName (or PageName | SiteName, etc.). Next.js supports this through a more advanced title configuration. 

// app/layout.js
export const metadata = {
  title: {
    template: "%s | The Wild Oasis",
    default: "Welcome | The Wild Oasis",
  },
};

üîé How it works

  • template uses %s as a placeholder.
  • If a page exports metadata.title, it replaces %s.
  • If a page does not export a title, Next.js uses default.

Example result:

  • Cabins page title ‚Üí Cabins | The Wild Oasis
  • Home page title (no override) ‚Üí Welcome | The Wild Oasis

4) üßæ Add a description (SEO)

The description becomes a meta name="description" entry in the document head. If you set it on the root layout, it becomes the default description for all pages (unless overridden). 

// app/layout.js
export const metadata = {
  title: {
    template: "%s | The Wild Oasis",
    default: "Welcome | The Wild Oasis",
  },
  description:
    "Luxurious cabin hotel in the Italian Dolomites, surrounded by mountains and forests.",
};

‚úÖ Beginner tip

If you don‚Äôt see the description in View Page Source, check the Elements tab instead. Some metadata is easier to confirm there depending on how the app is rendered.

5) ‚≠ê Favicon (tab icon) the Next.js way

Traditionally, favicons require manual <link rel="icon" ...> tags. In Next.js App Router, you can often do this with a convention-based file.

‚úÖ Convention

Put a file named icon in the app folder (e.g., app/icon.png). Next.js will use it as the favicon/app icon. 

// Example structure
app/
  icon.png      // ‚úÖ Next.js picks this up automatically
  layout.js
  page.js
public/
  ...

üìù Notes

  • The file can be PNG (useful for transparency), ICO, etc.
  • In the lecture, they copied an existing logo image, renamed it to icon.png, and placed it in app/.
  • No extra code was needed‚ÄîNext.js handled it.

7) ‚ö†Ô∏è Common pitfalls

  • Wrong file name: for favicon/app icon, the convention matters. Use icon.png (or the correct supported name) in app/.
  • Expecting everything in ‚ÄúView Page Source‚Äù: some metadata is easier to confirm in Elements.
  • Cache confusion: titles/icons may appear to ‚Äúlag‚Äù due to caching. Try a hard refresh if needed.
  • Overriding unintentionally: a page exporting metadata may override what you expect from the root layout.

‚úÖ Practical guideline
