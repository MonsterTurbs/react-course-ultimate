üéØ Goal & why it matters

Authentication (Auth.js) tells you who the user is (name/email/avatar). But your app still needs an internal record in your database. In this project, that internal record is a Guest row in Supabase.

Why do we need a Guest row?
  • üìå To store profile fields you don‚Äôt get from Google (e.g., National ID, nationality, flag).
  • üßæ To associate reservations/bookings to the correct person using guestId.
  • üîÅ To reuse the same Guest profile every time the user logs in.

Key idea: On first login, create a Guest in Supabase. On every session, attach guestId to the session so the app can use it everywhere.

‚úÖ Practical rule: email is how you find the guest; id is how you link records (bookings, profile updates, etc.).

üß≠ High-level flow (what happens on sign-in)

Sign-in flow overview (simplified):

  1. üë§ User authenticates with Google via Auth.js.
  2. üß™ Auth.js calls your signIn callback (before login finalizes).
  3. üîé In signIn: look up guest by email in Supabase.
  4. üÜï If not found: create guest row (await it!).
  5. ‚úÖ Return true to continue login, false to block.
  6. üß© Auth.js calls your session callback when building the session.
  7. üè∑Ô∏è In session: fetch guest again and attach guestId onto session.user.
‚ö†Ô∏è Why fetch again in session?
The signIn callback happens before the session object is finalized. The session callback is the right moment to safely enrich the session.

üß† Callback 1: signIn (create guest if missing)

You add a signIn callback in your Auth.js config to run logic each time someone signs in. Your job is to:

  • üîé Check if a guest already exists for user.email.
  • üÜï If not, create a new guest row with the minimum info you have.
  • ‚úÖ Return true if everything is okay, otherwise false.

üì¶ Example: data-service helpers (conceptual)

Your app typically has service functions like getGuest(email) and createGuest(data). If no guest exists, getGuest returns null.

‚úÖ Example: signIn callback

// In your Auth.js config (callbacks)
callbacks: {
  async signIn({ user, account, profile }) {
    try {
      const existingGuest = await getGuest(user.email);

      // If guest doesn't exist yet, create one
      if (!existingGuest)
        await createGuest({
          email: user.email,
          fullName: user.name,
          // nationalID, nationality, countryFlag can be filled later by the user
        });

      return true; // allow sign-in
    } catch (err) {
      // Something failed (DB down, permissions, etc.)
      return false; // block sign-in
    }
  },
}
üß® Critical: always await createGuest(...).
Without await, Auth.js may proceed to the session callback before the guest row is created, causing the next step to fail.

üè∑Ô∏è Callback 2: session (attach guestId to session)

The session callback runs after sign-in and whenever the session is accessed (e.g., when you call auth()). This is the ideal place to enrich the session with your internal database IDs.

‚úÖ What we want the session to look like

Instead of only:
  • session.user.name, session.user.email, session.user.image
We also want:
  • session.user.guestId ‚úÖ

‚úÖ Example: session callback

// In your Auth.js config (callbacks)
callbacks: {
  // ... signIn above

  async session({ session, user }) {
    // Find guest by the authenticated user's email
    const guest = await getGuest(session.user.email);

    // Attach guestId to the session so the app can use it everywhere
    session.user.guestId = guest.id;

    return session; // IMPORTANT
  },
}
‚úÖ Result: any server component can call const session = await auth() and immediately access session.user.guestId.

  • ‚úçÔ∏è Update profile fields (nationality, national ID, etc.) for the logged-in guest.

üõ†Ô∏è Common pitfalls & fixes

1) Forgot to await createGuest

Symptom: session callback can‚Äôt find the guest right after first sign-in.

Fix: await createGuest(...) inside signIn.

2) Didn‚Äôt return session in session callback

Symptom: calling auth() returns nothing / session becomes broken.

Fix: return session; at the end.

3) Unhandled errors inside callbacks

Error boundaries in your UI won‚Äôt catch these (they are part of auth flow).

Fix: use try/catch in signIn and return false if needed.

4) Assuming guestId exists everywhere automatically

You only get guestId in places where you call auth() (server) or use the client session APIs.

Fix: standardize session access via auth() on server components and pass required data down as props.
