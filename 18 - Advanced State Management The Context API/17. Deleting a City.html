1) Feature Overview (What we‚Äôre building) üß©

  • Each city item has a delete button (trash icon).
  • When the button is clicked, we should not navigate to the city details page.
  • We delete the city in two places:
    • üåê API: send an HTTP DELETE request to /cities/:id
    • üß† UI state: remove the city from the cities array in context (so list updates instantly)

Why this is ‚Äúfast‚Äù: deleting is usually just ‚Äúcall API + filter state‚Äù.

2) Problem: Clicking the delete button also clicks the Link üòµ

In CityItem, the whole item is typically a <Link>. If your delete button is inside that link, clicking the button triggers:

  • ‚úÖ the button click
  • ‚ùå the link navigation (because the click bubbles / default behavior happens)

Fix: prevent the link navigation ‚úÖ

Create a handler for the delete button and call e.preventDefault() (and optionally e.stopPropagation()).

// CityItem.jsx (concept)
function handleClick(e) {
  e.preventDefault();
  // optional but often helpful:
  // e.stopPropagation();

  console.log("Deleting...");
}

Common confusion: preventDefault() stops the default navigation behavior of the link. If your Link uses click handlers, stopPropagation() can help avoid triggering parent handlers.

3) Add the deleteCity function in Cities Context üèóÔ∏è

Because this is ‚Äúcities data,‚Äù we implement deletion inside the Cities Context Provider (same place you created getCity and createCity).

What deleteCity needs to do

  1. Send DELETE request to the API: ${BASE_URL}/cities/${id}
  2. Update UI state: remove that city from cities
// CitiesContext.jsx (concept)
async function deleteCity(id) {
  try {
    setIsLoading(true);

    await fetch(`${BASE_URL}/cities/${id}`, {
      method: "DELETE",
    });

    // UI state update: deleting = filtering
    setCities((cities) => cities.filter((city) => city.id !== id));
  } catch (err) {
    console.error("There was an error deleting city", err);
  } finally {
    setIsLoading(false);
  }
}

Mental model: DELETE mutates remote state; filter mutates UI state (immutably) so React re-renders immediately.

4) Expose deleteCity via context + use it in CityItem üîå

Make sure deleteCity is included in the provider‚Äôs value.

// CitiesContext.jsx (concept)
const value = {
  cities,
  isLoading,
  currentCity,
  getCity,
  createCity,
  deleteCity,

  <CitiesContext.Provider value={value}>

  </CitiesContext.Provider>

Then in CityItem (or wherever the delete button lives), read it from your custom hook:

// CityItem.jsx (concept)
import { useCities } from "../context/CitiesContext";

function CityItem({ city }) {
  const { deleteCity, currentCity } = useCities();

  function handleClick(e) {
    e.preventDefault();
    deleteCity(city.id);
  }

  return (
    <Link to={`${city.id}?lat=${city.position.lat}&lng=${city.position.lng}`}>
      <button onClick={handleClick} title="Delete">üóëÔ∏è</button>
      <span>{city.emoji} {city.cityName}</span>
    </Link>
  );
}

UX note: The global isLoading may gray out the entire list while deleting. That‚Äôs acceptable for a small app, but in bigger apps you may want per-item loading (e.g., deletingId).
