Goal of this lecture

We build the static layout of the main app screen: Sidebar on the left + Map on the right. üó∫Ô∏è

‚ÄúStatic‚Äù means: layout + components are on screen, but data/features (cities list, real map logic, etc.) can be added later.

Key idea ‚úÖ

Keep pages clean (mostly layout + composition), and push most markup/details into reusable components.

Big picture structure

Target UI layout üß©
  • AppLayout page: wraps everything in a flex container (or grid).
  • <Sidebar />: contains Logo, AppNav, City list placeholder, and a footer/copyright.
  • <Map />: placeholder map container on the right.
Folder organization üìÅ

You can keep a flat components folder (simple), or create a folder per component (component + module + tests together). In this course, we keep it flat.

Implementation steps

1) Move starter components into your components folder üì¶

  • Take all starter components and place them into src/components.
  • After moving, delete the now-unneeded starter folder.

2) Create Sidebar.jsx üß±

The sidebar uses CSS Modules: Sidebar.module.css already exists and provides the class sidebar.

// src/components/Sidebar.jsx
import styles from "./Sidebar.module.css";
import Logo from "./Logo";
import AppNav from "./AppNav";

export default function Sidebar() {
  return (
    <div className={styles.sidebar}>
      <Logo />
      <AppNav />

      <p>List of cities</p>

      <footer className={styles.footer}>
        <p className={styles.copyright}>
          &copy; {new Date().getFullYear()} by WorldWise Inc.
        </p>
      </footer>
    </div>
  );
}
Why compute the year like that? üóìÔ∏è

It avoids manual updates every year, and it‚Äôs harmless at runtime.

3) Create Map.jsx üó∫Ô∏è

For now, it‚Äôs only a placeholder with correct styling hooks.

// src/components/Map.jsx
import styles from "./Map.module.css";

export default function Map() {
  return <div className={styles.mapContainer}>Map</div>;
}

4) Compose them in AppLayout.jsx üß≠

AppLayout should be ‚Äúthin‚Äù: apply layout styles and render the main components.

// src/pages/AppLayout.jsx
import styles from "./AppLayout.module.css";
import Sidebar from "../components/Sidebar";
import Map from "../components/Map";

export default function AppLayout() {
  return (
    <div className={styles.app}>
      <Sidebar />
      <Map />
    </div>
  );
}

The layout works because styles.app is typically a flex container: sidebar on left, map on right.

5) Confirm styling is applied ‚úÖ

  • If the layout looks ‚Äústacked‚Äù (sidebar above map), likely styles.app isn‚Äôt applied or the CSS module import is wrong.
  • If sidebar styles don‚Äôt apply, verify: className={styles.sidebar} and the correct module file name.

VS Code snippet: quick CSS module import

To speed up workflow, add a snippet that auto-imports the correct .module.css file. ‚ö°

What the snippet does
  • When you type csm and hit enter, it inserts: import styles from "./FileName.module.css"
  • It uses the current file name automatically.
// Suggested snippet body (conceptual)
// Prefix: csm
// Result:
import styles from "./$TM_FILENAME_BASE.module.css";

In the lecture, the snippet is stored in VS Code user snippets and triggered via prefix CSM.

Patterns and best practices

  • ‚úÖ Page = composition: Keep pages mostly for wiring components together.
  • ‚úÖ Component owns its styling: one .module.css per component.
  • ‚úÖ Use semantic HTML: using <footer> is fine even inside a sidebar.
  • ‚úÖ Prefer predictable imports: avoid overly clever destructuring of CSS module classes if it reduces clarity.

Common pitfalls

  • ‚ö†Ô∏è Wrong CSS module import path: e.g., importing from pages instead of components, or vice versa.
  • ‚ö†Ô∏è CSS module file name mismatch: must be ComponentName.module.css (case-sensitive on many systems).
  • ‚ö†Ô∏è Missing layout class: forgetting className={styles.app} in AppLayout will break the two-column layout.
  • ‚ö†Ô∏è Over-stuffing pages: putting all sidebar markup directly in the page makes the codebase harder to scale.
